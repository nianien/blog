17:Ta73e,<p>LLM 的训练目标是 next-token prediction，这种机制天然缺两种能力——<strong>前瞻</strong>（生成第一步时不会考虑第五步的后果）和<strong>回溯</strong>（一旦写下就不能回头修正）。给 LLM 补上这两个缺口的工程手段有三种：规划补前瞻、推理补深度、反思补回溯。三者不是&quot;加上模型就有&quot;，是用 Prompt、Schema、终止条件和评估反馈搭出来的外部脚手架。下面展开四种规划范式的伪代码骨架、反思机制的 schema 与终止条件，以及在 o1/o3 这类推理模型出现后，&quot;什么时候该升级到哪一档&quot;的判断信号怎么变。</p>
<hr>
<h2>1. 为什么 LLM 缺这三种能力</h2>
<p>LLM 的训练目标是 next-token prediction——给定前文预测下一个 token。这种机制天然缺两种能力：</p>
<table>
<thead>
<tr>
<th>缺失能力</th>
<th>表现</th>
</tr>
</thead>
<tbody><tr>
<td><strong>前瞻</strong>（Lookahead）</td>
<td>生成第一步时不会考虑&quot;这个决定在第五步会导致什么后果&quot;——每步选局部最优，但局部最优叠加不等于全局最优</td>
</tr>
<tr>
<td><strong>回溯</strong>（Backtrack）</td>
<td>一旦生成就不回头修正。中间出错，后续 token 也基于错误前提继续生成</td>
</tr>
</tbody></table>
<p>三种能力分别补这两个缺：</p>
<table>
<thead>
<tr>
<th>能力</th>
<th>补什么</th>
<th>在什么阶段</th>
</tr>
</thead>
<tbody><tr>
<td><strong>规划</strong></td>
<td>前瞻</td>
<td>执行前——把大目标拆成子目标，考虑步骤间依赖</td>
</tr>
<tr>
<td><strong>推理</strong></td>
<td>思考深度</td>
<td>每步决策前——通过结构化思维链提升单步质量</td>
</tr>
<tr>
<td><strong>反思</strong></td>
<td>回溯</td>
<td>执行后——检查结果、分析错误、决定重试或调整</td>
</tr>
</tbody></table>
<p>三者结合，Agent 才能从&quot;工具调用器&quot;变成&quot;问题解决者&quot;。规划解决&quot;做什么&quot;，推理解决&quot;想清楚&quot;，反思解决&quot;对不对&quot;。</p>
<p><strong>这三种能力的边界其实容易混淆</strong>。规划和推理都在 LLM 调用内部完成，但规划关心的是&quot;目标到步骤&quot;的分解，推理关心的是&quot;前提到结论&quot;的演绎。反思则发生在执行之后，关心&quot;实际结果是否符合预期&quot;。一个常见的设计错误是把三者混在一个 prompt 里——让 LLM 同时规划、推理、反思——结果三个能力都做得不好。结构化的做法是把三者拆成不同的调用阶段，让每次调用只承担一个职责。</p>
<hr>
<h2>2. 规划范式的四种主流形态</h2>
<p><img src="/images/blog/agentic/paradigm-evolution.svg" alt="范式演进"></p>
<table>
<thead>
<tr>
<th>范式</th>
<th>思想</th>
<th>解决什么</th>
<th>引入什么问题</th>
</tr>
</thead>
<tbody><tr>
<td><strong>ReAct</strong></td>
<td>每步交替 Thought 和 Action，下一步基于上一步 Observation</td>
<td>多步推理 + 行动</td>
<td>Greedy，缺全局视野</td>
</tr>
<tr>
<td><strong>Plan-and-Execute</strong></td>
<td>先规划完整计划，再逐步执行</td>
<td>全局视野、可追踪、可并行</td>
<td>计划可能过时，修正成本高</td>
</tr>
<tr>
<td><strong>Tree-of-Thought</strong></td>
<td>同时考虑多条推理路径，评估后选最优深入</td>
<td>探索多种可能性</td>
<td>成本 8-10 倍于 ReAct</td>
</tr>
<tr>
<td><strong>Hierarchical</strong></td>
<td>递归分解——高层拆子目标，低层拆具体动作</td>
<td>处理跨粒度的真正复杂任务</td>
<td>架构复杂，调试困难</td>
</tr>
</tbody></table>
<h3>2.1 ReAct：贪心但实用</h3>
<p>ReAct 的设计哲学是&quot;边走边看&quot;：每一步基于当前观察决定下一个动作，不预先规划全部步骤。优势是<strong>应变能力强</strong>——如果第 2 步发现新信息推翻了第 1 步的假设，ReAct 在第 3 步就能调整方向；劣势是<strong>没有全局视图</strong>——可能在某条死胡同里走很远才意识到该回头。</p>
<p>最小骨架就是一个 while 循环：</p>
<pre><code class="language-python">def react_loop(goal: str, tools: list[Tool], max_steps: int = 10) -&gt; str:
    &quot;&quot;&quot;ReAct：Thought → Action → Observation 反复直到 final answer&quot;&quot;&quot;
    messages = [system_msg(REACT_PROMPT), user_msg(goal)]

    for step in range(max_steps):
        # 一步 Think + Plan：LLM 输出 thought 和可选的 tool_call
        response = llm.complete(messages, tools=tools)

        # 如果没有 tool_call，说明 LLM 认为已能给出最终答案
        if not response.tool_calls:
            return response.text

        # 否则执行工具，结果作为下一轮的 observation
        for tc in response.tool_calls:
            observation = invoke(tc)
            messages.append(tool_msg(tc.id, observation))

    return safe_terminate(&quot;max_steps reached&quot;)
</code></pre>
<p>工程上 ReAct 几乎是默认选择，因为它的实现成本极低——一个 while 循环加 Thought-Action-Observation 三段式 prompt 就够了。绝大多数生产 Agent（Cursor、Claude Code、客服 Agent）的核心循环都是 ReAct 的变种。</p>
<p><strong>ReAct 的成本陷阱</strong>：每步 LLM 调用，上下文每步累积，token 消耗 O(N²)。5 步约 15K-25K tokens，10 步可能 50K+。这意味着 ReAct 不适合超过 7-8 步的长任务——再长就要考虑 Plan-and-Execute 或 Hierarchical。</p>
<h3>2.2 Plan-and-Execute：先想后做</h3>
<p>Plan-and-Execute 把规划和执行解耦——先用一次 LLM 调用产出完整计划（步骤列表 + 依赖关系），然后逐步执行：</p>
<pre><code class="language-python">def plan_and_execute(goal: str, tools: list[Tool]) -&gt; str:
    # 阶段 1：一次 LLM 调用产出完整 Plan（强 schema 约束）
    plan: Plan = llm.complete(
        prompt=PLANNER_PROMPT.format(goal=goal, tools=tools),
        schema=PLAN_SCHEMA,
    )

    # 阶段 2：按依赖关系执行 Plan
    results = {}
    for step in topological_sort(plan.steps):
        # 注入前置步骤的结果作为本步的上下文
        ctx = {dep_id: results[dep_id] for dep_id in step.depends_on}
        results[step.id] = execute_step(step, ctx, tools)

        # 关键：每步执行后判断 Plan 是否仍然有效
        if replan_signal(results[step.id], plan):
            plan = llm.complete(
                prompt=REPLAN_PROMPT.format(plan=plan, executed=results),
                schema=PLAN_SCHEMA,
            )

    # 阶段 3：最终综合
    return llm.complete(SYNTHESIS_PROMPT.format(plan=plan, results=results))
</code></pre>
<p>Plan 本身需要严格 schema，否则下游执行器没法消费：</p>
<pre><code class="language-json">{
  &quot;name&quot;: &quot;task_plan&quot;,
  &quot;parameters&quot;: {
    &quot;type&quot;: &quot;object&quot;,
    &quot;properties&quot;: {
      &quot;analysis&quot;: {
        &quot;type&quot;: &quot;string&quot;,
        &quot;description&quot;: &quot;简短分析任务复杂度与拆分思路（≤ 100 字）&quot;
      },
      &quot;steps&quot;: {
        &quot;type&quot;: &quot;array&quot;,
        &quot;items&quot;: {
          &quot;type&quot;: &quot;object&quot;,
          &quot;properties&quot;: {
            &quot;id&quot;: {&quot;type&quot;: &quot;integer&quot;},
            &quot;description&quot;: {&quot;type&quot;: &quot;string&quot;},
            &quot;tool&quot;: {&quot;type&quot;: &quot;string&quot;, &quot;enum&quot;: [&quot;search&quot;, &quot;calc&quot;, &quot;summarize&quot;, null]},
            &quot;args&quot;: {&quot;type&quot;: &quot;object&quot;},
            &quot;depends_on&quot;: {&quot;type&quot;: &quot;array&quot;, &quot;items&quot;: {&quot;type&quot;: &quot;integer&quot;}},
            &quot;expected_output&quot;: {&quot;type&quot;: &quot;string&quot;},
            &quot;fallback_action&quot;: {&quot;type&quot;: &quot;string&quot;}
          },
          &quot;required&quot;: [&quot;id&quot;, &quot;description&quot;, &quot;depends_on&quot;]
        },
        &quot;minItems&quot;: 3,
        &quot;maxItems&quot;: 7
      },
      &quot;synthesis_step_id&quot;: {&quot;type&quot;: &quot;integer&quot;}
    },
    &quot;required&quot;: [&quot;analysis&quot;, &quot;steps&quot;, &quot;synthesis_step_id&quot;]
  }
}
</code></pre>
<p><code>minItems: 3, maxItems: 7</code> 是工程上很有效的约束——少于 3 步通常是&quot;该用 ReAct 不该用 Plan-Exec&quot;，多于 7 步通常是&quot;过度拆分&quot;。<strong>Plan-and-Execute 的隐藏价值</strong>：当任务超过 6-7 步时，通常比 ReAct 省钱。原因是 ReAct 每步都要重新把所有历史塞进 prompt，而 Plan-and-Execute 的执行步只需要塞当前步骤的局部上下文。</p>
<p>Replan 机制是这个范式的关键设计点。常见做法是：执行每步后让 LLM 判断&quot;当前计划是否仍然有效&quot;——如果有效则继续，无效则触发 Replan。Replan 的代价不便宜（要重新做一次完整规划），所以触发频率要控制——不能每步都 Replan，但也不能完全不 Replan。</p>
<h3>2.3 Tree-of-Thought：探索多条路径</h3>
<p>ToT 的思想来自经典搜索算法——把推理建模成树搜索，每个节点是一个中间状态，分支是可能的下一步。LLM 同时生成多个候选下一步、对每个候选评分、选评分最高的展开：</p>
<pre><code class="language-python">def tot_search(
    task: str,
    branching_factor: int = 3,
    max_depth: int = 4,
    beam_width: int = 2,
) -&gt; str:
    &quot;&quot;&quot;Tree-of-Thought：beam search 风格的多路径推理&quot;&quot;&quot;
    beam = [Node(state=task, score=0.0, parent=None)]

    for depth in range(max_depth):
        candidates = []
        for node in beam:
            # 每个节点生成 N 个候选下一步
            children = llm.complete(
                EXPAND_PROMPT.format(state=node.state),
                n=branching_factor,
            )
            # 对每个候选独立打分（用同一个或不同的 LLM）
            scored = [
                Node(state=c, score=llm_judge(c, task), parent=node)
                for c in children
            ]
            candidates.extend(scored)

        # 剪枝：只保留 top beam_width 个继续展开
        beam = sorted(candidates, key=lambda n: -n.score)[:beam_width]

        if any(n.is_terminal for n in beam):
            return best(beam).reconstruct_path()

    return best(beam).reconstruct_path()
</code></pre>
<p>ToT 适合<strong>结果可评估</strong>的任务——数学题（答案可验证）、代码生成（能跑测试）、创意写作（可由人或另一个 LLM 评分）。不适合<strong>结果难评估</strong>的任务，因为 ToT 的核心是评分驱动剪枝，评分不准就退化成&quot;昂贵的随机搜索&quot;。</p>
<p>成本是 ToT 的硬伤。一个分支宽度 3、深度 4 的搜索树，理论上要做 3⁴=81 次 LLM 调用。即使做剪枝，5-10 倍的成本相比 ReAct 也是常态。</p>
<p><strong>什么时候 ToT 的 5-10 倍成本值得花</strong>？三个判据，全部满足才上：</p>

<h3>2.4 Hierarchical：递归分解</h3>
<p>Hierarchical 处理的是&quot;跨粒度复杂度&quot;——任务大到单个 LLM 调用根本理解不了的时候。&quot;实现一个完整的电商系统&quot;这种任务，没法直接 ReAct（步骤可能上千）、也没法 Plan-and-Execute（一个计划塞不下）。解法是递归：</p>
<pre><code class="language-python">def hierarchical_solve(goal: str, depth: int = 0, max_depth: int = 3) -&gt; Result:
    &quot;&quot;&quot;递归分解：顶层拆子目标，每个子目标交给下层 Agent&quot;&quot;&quot;
    if depth &gt;= max_depth or is_atomic(goal):
        # 触底——用 ReAct 直接解决最细粒度任务
        return react_loop(goal, tools=available_tools())

    # 顶层只做分解，不做执行
    sub_goals = llm.complete(
        DECOMPOSE_PROMPT.format(goal=goal, depth=depth),
        schema=DECOMPOSITION_SCHEMA,
    )

    # 递归解决每个子目标
    sub_results = []
    for sub in sub_goals:
        sub_result = hierarchical_solve(sub.description, depth + 1, max_depth)
        sub_results.append((sub, sub_result))

        # 子目标失败要决定是否回到上层重规划
        if sub_result.failed and sub.is_critical:
            return Result.failed(reason=f&quot;critical sub-goal failed at depth {depth+1}&quot;)

    # 上层 Agent 汇总下层结果
    return llm.complete(
        AGGREGATE_PROMPT.format(goal=goal, sub_results=sub_results),
    )
</code></pre>
<p>Hierarchical 的工程难点是<strong>层间通信</strong>：上层 Agent 要把任务&quot;传&quot;给下层，下层 Agent 要把结果&quot;返回&quot;给上层。如果传递的信息过多，上层 Agent 上下文会爆炸；过少则下层 Agent 缺失关键约束。这是为什么 Hierarchical 和多 Agent 协作里的 Supervisor-Worker 模式高度同构——本质上是同一个问题。</p>

<h2>3. 如何让 LLM 真的会规划</h2>
<p>ReAct/Plan-and-Execute 都依赖一个前提：<strong>LLM 能生成合理的计划</strong>。但 LLM 不是天生会规划的——训练目标是预测下一个 token，不是&quot;生成最优任务分解&quot;。规划能力需要 prompt 引导。</p>
<p>一个合格的规划 Prompt 的骨架：</p>
<pre><code class="language-text">你是任务规划者，把一个高层目标拆解为可执行的步骤列表。

## 可用工具
{tools_list}                       # 明确列出，LLM 不可发明新工具

## 输入
目标：{goal}
约束：{constraints}                # 时间预算、成本预算、安全要求

## 规划要求
1. 先输出 analysis 字段判断任务是否需要规划
   - 简单到 1-2 步能做完的，标 needs_planning=false，直接返回 1 步计划
2. 步骤数控制在 3-7 步之间，超出意味着过度拆分
3. 每步必须能用列表中的工具完成；如果工具不够，明确说&quot;need_tool: X&quot;
4. 标注 depends_on——只允许引用更小 id 的步骤（避免循环依赖）
5. 标注 expected_output 强迫思考步骤间数据流
6. 最后一步必须是 synthesis（tool=null），不能以工具调用结尾
7. 每步标注 fallback_action：失败时的兜底动作

输出严格遵循 JSON Schema：{plan_schema_json}
</code></pre>

<thead>
<tr>
<th>要素</th>
<th>缺失后果</th>
</tr>
</thead>
<tbody><tr>
<td>角色定义（&quot;你是任务规划者&quot;）</td>
<td>LLM 直接尝试回答问题，跳过规划</td>
</tr>
<tr>
<td>可用资源（工具列表）</td>
<td>生成不可执行的步骤（引用不存在的工具）</td>
</tr>
<tr>
<td>输出格式约束（JSON Schema）</td>
<td>生成自由文本&quot;计划&quot;，下游无法解析</td>
</tr>
<tr>
<td>规划质量约束（步骤数、依赖标注）</td>
<td>过度拆分或步骤间逻辑断裂</td>
</tr>
</tbody></table>
<p>三个反复有效的技巧：</p>
<ul>
<li><strong>强制 analysis 字段</strong>：让 LLM 先回答&quot;这任务是否需要多步规划&quot;，避免给简单问题写 10 步计划</li>
<li><strong>强制 expected_output</strong>：每步预估输出，迫使 LLM 思考步骤间数据流动，减少&quot;步骤 3 依赖步骤 2 的结果，但步骤 2 根本不产生这个结果&quot;</li>
<li><strong>限制步骤数为 3-7 步</strong>：没有上限时 LLM 倾向于过度拆分，每步做的事微不足道，反而增加失败概率</li>
</ul>

<h2>4. 推理增强：让 LLM 想得更深</h2>
<p>规划解决&quot;做事的方法论&quot;，推理增强解决&quot;每步思考的深度&quot;——LLM 在某一步推理本身错了，规划再好也救不了。</p>
<h3>4.1 Chain-of-Thought 家族</h3>
<p>CoT 的核心思想极简单：让 LLM 在输出最终答案前先输出中间推理步骤。一句 &quot;Let&#39;s think step by step&quot; 在数学和逻辑推理任务上能让准确率提升 10-30%。但 CoT 不是免费的——它把 token 消耗增加 2-5 倍，并且延迟相应增加。</p>

<h3>4.2 Self-Consistency 的伪代码</h3>
<pre><code class="language-python">def self_consistency(
    question: str,
    n_samples: int = 5,
    temperature: float = 0.7,
) -&gt; str:
    &quot;&quot;&quot;同一问题独立推理 N 次，投票取多数答案&quot;&quot;&quot;
    answers = []
    for _ in range(n_samples):
        # 高温度让 LLM 走不同的推理路径
        cot = llm.complete(COT_PROMPT.format(q=question), temperature=temperature)
        # 从 CoT 末尾提取最终答案（按 schema 强制结构）
        final = extract_final_answer(cot)
        answers.append(final)

    # 取出现频率最高的答案
    counts = Counter(answers)
    winner, freq = counts.most_common(1)[0]

    # 一致性低（freq &lt; n/2）意味着 LLM 自己也没把握
    if freq &lt; n_samples / 2:
        return escalate(&quot;low_self_consistency&quot;, candidates=counts)

    return winner
</code></pre>

<p>判断要不要上 Self-Consistency 的最简公式：<strong>单次错误代价 ÷ N 倍 LLM 调用成本 &gt; 1，就划算</strong>。多次独立推理不容易都犯同一个错——这是 Self-Consistency 的统计基础。</p>
<p><strong>CoT 的失效场景</strong>：当任务本身是&quot;模式识别&quot;而非&quot;逻辑推演&quot;时，CoT 不仅不增益，还可能损害准确率——LLM 被迫&quot;编造&quot;推理过程，反而引入错误。情感分析、分类、命名实体识别这类任务通常不需要 CoT。</p>
<h3>4.3 推理时扩展（Test-time Compute Scaling）</h3>
<p>2024-2025 推理领域最重要的发现：<strong>推理阶段投入更多算力可以持续提升质量，且幅度可预测</strong>。OpenAI 的 o1/o3 和 Anthropic 的 Extended Thinking 都是这个思路的产品化。这种&quot;推理算力&quot;既包括模型内部的隐式思考（reasoning tokens），也包括 Self-Consistency、ToT 这类显式多路径推理。</p>
<p>对 Agent 工程师，核心问题是：<strong>什么时候值得让模型多想一会儿</strong>？</p>

<p><strong>有效模式</strong>：用推理模型做规划（一次性高质量规划），用标准模型做执行（快速逐步执行）——和分层规划同构。这种&quot;贵模型规划 + 便宜模型执行&quot;的混合架构在生产中越来越常见，原因是规划阶段的质量天花板很高（一个好规划能省下后面所有步骤），而执行阶段的边际收益很低（执行 prompt 通常简单）。</p>
<p>另一个值得注意的变化：<strong>推理模型让显式规划的价值下降了</strong>。以前 Plan-and-Execute 的核心价值之一是&quot;用一次完整规划替代多次 ReAct&quot;，但推理模型内部已经做了类似的隐式规划，所以推理模型 + ReAct 的组合在很多场景下可以达到&quot;普通模型 + Plan-and-Execute&quot;的效果。</p>

<h2>5. 反思：自我纠错</h2>
<p>Agent 三类典型失败——LLM 输出错误（幻觉、逻辑错）、工具执行失败（超时、参数错）、计划不可行（前提假设不成立）。没有反思，错误<strong>无意识地传播</strong>——第 2 步的错变成第 3 步的输入，错误累积。</p>
<p>反思的核心问题：<strong>用什么评估、什么时候停</strong>。</p>
<h3>5.1 谁来评估：Self-Critique 的局限</h3>
<p>让同一个 LLM 评估自己的输出。理论支持是 LLM 在<strong>验证</strong>上通常比<strong>生成</strong>更强——就像检查别人的代码比自己写更容易。这个理论部分成立——在某些任务上 LLM 验证准确率确实高于生成准确率，但前提是验证标准清晰且可检查（&quot;这段 SQL 是否能跑通&quot;）。</p>
<p>但盲区在于：<strong>LLM 的系统性偏见在生成和评估中是一致的</strong>——一个倾向于过度自信的模型，评估自己的输出时也倾向于给高分。如果生成阶段 LLM 漏掉了某种边缘情况，评估阶段它大概率也漏掉，因为&quot;想到这个边缘情况&quot;本身是它的盲点。</p>
<p>工程上的对策有几种：</p>

</tr>
<tr>
<td>多次自评 + 投票</td>
<td>Self-Consistency 思路</td>
<td>成本翻几倍</td>
</tr>
<tr>
<td>显式 checklist</td>
<td>列出&quot;评估清单&quot;强制 LLM 逐项检查</td>
<td>prompt 工程负担</td>
</tr>
</tbody></table>
<p><strong>最有效的反思是外部信号驱动的</strong>——代码能编译运行、SQL 能查到数据、API 返回成功状态码——这些是 LLM 无法&quot;自欺欺人&quot;的硬信号。能用外部信号就别用 LLM Self-Critique。</p>
<h3>5.2 反思的结构化输出</h3>
<p>为什么反思必须是结构化输出而不是自由文本？三个理由：<strong>自由文本的反思下游没法消费</strong>——代码不能 <code>if reflection.contains(&quot;应该重试&quot;)</code>，必须 <code>if reflection.decision == &quot;retry&quot;</code>；<strong>自由文本会鼓励 LLM 写看起来很有道理的废话</strong>，结构化 schema 强迫它在固定字段里给出可被检查的判断；<strong>自由文本无法做反思的反思</strong>——下一轮想知道上轮反思的根因/置信度/改进幅度，只有字段化才能查询。一句话：<strong>反思的输出格式决定了反思机制能不能闭环</strong>。</p>
<p>Reflect 的 schema 决定了下游能否消费它的判断。一个最小可用的 schema：</p>
<pre><code class="language-json">{
  &quot;name&quot;: &quot;reflection&quot;,
  &quot;parameters&quot;: {
    &quot;type&quot;: &quot;object&quot;,
    &quot;properties&quot;: {
      &quot;what_went_well&quot;: {
        &quot;type&quot;: &quot;array&quot;,
        &quot;items&quot;: {&quot;type&quot;: &quot;string&quot;}
      },
      &quot;what_went_wrong&quot;: {
        &quot;type&quot;: &quot;array&quot;,
        &quot;items&quot;: {&quot;type&quot;: &quot;string&quot;}
      },
      &quot;root_cause&quot;: {
        &quot;type&quot;: &quot;string&quot;,
        &quot;description&quot;: &quot;不是症状描述，是根因——&#39;API key 错&#39; 而非 &#39;工具调用失败&#39;&quot;
      },
      &quot;what_to_do_next&quot;: {
        &quot;type&quot;: &quot;string&quot;,
        &quot;description&quot;: &quot;具体改进方向，能直接指导下一步动作&quot;
      },
      &quot;decision&quot;: {
        &quot;type&quot;: &quot;string&quot;,
        &quot;enum&quot;: [&quot;accept&quot;, &quot;retry_same&quot;, &quot;retry_with_change&quot;, &quot;escalate_to_human&quot;, &quot;give_up&quot;]
      },
      &quot;confidence&quot;: {&quot;type&quot;: &quot;number&quot;, &quot;minimum&quot;: 0, &quot;maximum&quot;: 1},
      &quot;expected_improvement&quot;: {
        &quot;type&quot;: &quot;number&quot;,
        &quot;description&quot;: &quot;预计下次重试能改善多少（0-1）。预计 &lt; 0.1 时不该再重试&quot;
      }
    },
    &quot;required&quot;: [&quot;root_cause&quot;, &quot;decision&quot;, &quot;confidence&quot;, &quot;expected_improvement&quot;]
  }
}
</code></pre>
<p>最关键的字段是 <strong>root_cause</strong>——它强迫 LLM 不要停留在&quot;输出有点不对&quot;这种症状层面，而要追溯到&quot;为什么会出这个错&quot;。没有 root_cause 字段的反思往往退化成&quot;复述错误&quot;，对下一轮决策没有指导价值。</p>
<p><strong>expected_improvement 是改善 improvement_threshold 的关键设计</strong>——下面会展开。</p>
<h3>5.3 终止条件：四道防线</h3>
<p>反思不能无限循环。任何一个条件触发都终止：</p>
<p><img src="/images/blog/agentic/reflection-stop-condition.svg" alt="反思终止条件"></p>
<pre><code class="language-python">def should_stop_reflecting(history: list[Reflection], state: ReflectState) -&gt; bool:
    &quot;&quot;&quot;四道防线，任一触发就停&quot;&quot;&quot;
    latest = history[-1]

    # 防线 1：质量达标，正常退出
    if latest.confidence &gt;= 0.75 and latest.decision == &quot;accept&quot;:
        return True

    # 防线 2：重试次数上限
    if len(history) &gt;= 3:
        return True

    # 防线 3：连续两轮改善幅度太小——反思没有意义了
    if len(history) &gt;= 2:
        if latest.expected_improvement &lt; 0.1:
            return True
        # 实际改善（用外部信号衡量）也很小
        if abs(state.last_score - state.previous_score) &lt; 0.05:
            return True

    # 防线 4：成本兜底
    if state.tokens_used &gt;= state.token_budget:
        return True

    return False
</code></pre>

<th>防什么</th>

<p><strong>最容易忽视的是 improvement_threshold</strong>——Agent 反复反思但不改进，是反思机制最常见的失败模式。表现形式是 LLM 每次反思都&quot;识别出新问题&quot;但其实是同一个根因的不同表述，重试再多次也解决不了。improvement_threshold 在工程上是说&quot;反思两轮还没明显进步就别再试了，要么人工介入要么换策略&quot;。</p>

<pre><code class="language-python">class ReflectMode(Enum):
    SKIP                   = &quot;skip&quot;                  # 不反思，直接返回结果
    EXTERNAL_ONLY          = &quot;external_only&quot;         # 只用外部信号（跑测试/查 API），不用 LLM
    SINGLE_SHOT_CRITIQUE   = &quot;single_shot_critique&quot;  # 一次 LLM 自评，无重试
    ONE_ROUND_LLM_JUDGE    = &quot;one_round_llm_judge&quot;   # 一轮 LLM judge，结构化决策
    FULL_REFLECT_LOOP      = &quot;full_reflect_loop&quot;     # 完整结构化反思 + 多轮重试

def select_reflection_mode(task: TaskSpec, budget: Budget) -&gt; ReflectMode:
    if task.error_cost == &quot;low&quot;:
        return ReflectMode.SKIP

    # 外部信号优先——比 LLM Self-Critique 强一个数量级
    if task.has_external_validator:
        return ReflectMode.EXTERNAL_ONLY  # 跑测试、查 API、校验 schema

    if not budget.allows_multi_round_reflection:
        return ReflectMode.SINGLE_SHOT_CRITIQUE

    if task.latency_sensitive:
        return ReflectMode.ONE_ROUND_LLM_JUDGE

    return ReflectMode.FULL_REFLECT_LOOP   # 完整结构化反思 + 多轮
</code></pre>

<h2>8. 三种能力如何同时上场</h2>
<p>规划、推理、反思单独使用都能提升 Agent 质量，但真正的杠杆在于<strong>三者协同</strong>：</p>
<table>
<thead>
<tr>
<th>协同模式</th>
<th>描述</th>
<th>适用</th>
</tr>
</thead>
<tbody><tr>
<td>规划期推理</td>
<td>规划时让 LLM 用 CoT 仔细思考分解方案</td>
<td>复杂任务，规划质量决定全局质量</td>
</tr>
<tr>
<td>执行后反思</td>
<td>每个执行步骤完成后用 Reflection 评估，错则触发局部 Replan</td>
<td>不可逆操作前的兜底</td>
</tr>
<tr>
<td>反思驱动重规划</td>
<td>多步反思发现整体计划走偏，触发完整 Replan</td>
<td>长任务的中段调整</td>
</tr>
<tr>
<td>推理驱动剪枝</td>
<td>ToT 用 CoT 评估每个分支，剪掉低分支</td>
<td>搜索空间大的任务</td>
</tr>

<p>一个把三者编织在一起的伪代码——核心是<strong>三档模型对应三种用途</strong>：贵推理模型管规划质量（一次性投入）、标准模型管执行速度（每步一次）、便宜小模型管反思评判（关键节点偶发）。</p>
<pre><code class="language-python">def agent_with_three_capabilities(goal: str) -&gt; str:
    plan = llm.complete(PLANNER_PROMPT.format(goal=goal),
                        model=&quot;reasoning&quot;, schema=PLAN_SCHEMA)   # 贵但只一次

    results = {}
    for step in topological_sort(plan.steps):
        results[step.id] = react_loop(step.description, model=&quot;standard&quot;)  # 标准模型 ReAct

        if step.is_irreversible:                                  # 关键节点才反思
            reflect = llm.complete(REFLECT_PROMPT.format(step=step, result=results[step.id]),
                                   model=&quot;cheap_judge&quot;, schema=REFLECTION_SCHEMA)
            if reflect.decision == &quot;retry_with_change&quot;: continue
            if reflect.decision == &quot;escalate_to_human&quot;: return human_approval(...)
            if reflect.decision == &quot;give_up&quot;:           return safe_fail(...)

        if accumulated_drift(results, plan) &gt; REPLAN_THRESHOLD:   # 整体走偏才重规划
            plan = replan(goal, plan, results)
    return synthesize(plan, results)
</code></pre>
<p>模型分档的依据：规划阶段的质量天花板高（一个好规划能省下后面所有步骤），所以贵模型一次性投入合算；执行阶段已知步骤要做什么，标准模型够用；反思阶段只是&quot;是否通过 + 决策标签&quot;，小模型 + 结构化 schema 即可。三档不区分会让账单贵 5-10 倍。</p>
<p>这种协同的工程意义是：<strong>不要把规划、推理、反思看成独立的功能模块，而要看成同一个循环里的不同阶段</strong>。一个生产级 Agent 几乎一定会同时用到三者，问题只是用得轻还是用得重。</p>
<hr>
<h2>9. 单 Agent 规划撞墙的四个地方</h2>
<p>规划、推理、反思让单 Agent 从&quot;走一步看一步&quot;进化到&quot;先想后做再检查&quot;。但单 Agent 的认知能力终有上限：</p>
<ul>
<li><strong>上下文窗口限制</strong>：任务涉及的知识超出 context window 时，再好的规划也没用</li>
<li><strong>专业性限制</strong>：一个 Agent 很难同时擅长编码、写作和数据分析——就像一个人很难同时是程序员、设计师和产品经理</li>
<li><strong>执行效率限制</strong>：单 Agent 串行执行，即使计划中的步骤可以并行</li>
<li><strong>决策路径单一</strong>：单 Agent 的决策由同一个 LLM 在同一个 prompt 下做出，缺少不同视角的对抗与互补</li>
</ul>
<p>当这些限制成为瓶颈，需要的不是更好的规划算法，而是<strong>多个 Agent 的协作</strong>——每个专注擅长领域，由 Orchestrator 协调。</p>
<hr>
<h2>10. 给 LLM 补三种能力的统一逻辑</h2>
<p>规划、推理、反思看起来是三件事，本质上是给 LLM 补同一件事——<strong>它在 next-token prediction 这个生成机制下做不到的非线性思考</strong>。规划补前瞻（任务还没开始就想清楚），推理增强补深度（每一步想得更透），反思补回溯（错了之后能找根因）。三者都不是&quot;换个更大的模型自然就有&quot;，是工程上用外部结构搭出来的——CoT 是结构、Plan 是结构、Self-Critique 是结构，让 LLM 在受控的脚手架里产出可消费的中间产物。</p>
<p>工程上最常见的判断错误是把这三件事都当成&quot;上模型就好&quot;。每升级一个模型确实会让基础推理强一些，但规划深度不会自动出现、反思的根因分析也不会自动严谨——这些都是 Prompt 工程 + 输出 Schema + 终止条件 + 评估反馈共同作用的产物。把它们当&quot;功能模块&quot;装上去就够用是浅层理解，把它们看成&quot;循环里的不同阶段&quot;是工程视角。</p>

<h2>1. Agent 为什么需要记忆</h2>
<p>LLM 是 stateless function——同一个输入两次调用各自独立，不知道上次发生过什么。这在单轮问答可以接受，但当 LLM 嵌入 Agent 后立刻成为致命缺陷：</p>
<ul>
<li>多轮对话：&quot;把上面那个改成蓝色&quot;——&quot;上面那个&quot;在哪里？</li>
<li>长任务执行：Agent 跑到第 5 步，需要第 2 步的输出来做决策</li>
<li>跨会话连续性：昨天分析过的报告，今天问&quot;上次的结论是什么？&quot;</li>
<li>个性化：用户偏好（简洁、表格、Markdown）需要记住</li>
</ul>
<p>工程上的根问题：在 LLM 这个 stateless function 之外，<strong>构建一套 stateful 的记忆系统</strong>，让 Agent 在有限的 context window 内获得&quot;无限&quot;的记忆能力。</p>
<p>这个问题在传统软件里有现成解法（数据库、缓存、文件系统），但 Agent 场景下有一个独特约束：<strong>最终所有信息都要塞进 LLM 的 context window</strong>。数据库里存了 100 万条记忆没用——LLM 一次能看到的只有 context window 那几万 token。所以记忆系统的工程难点不是&quot;如何存&quot;，而是&quot;如何在合适的时机把合适的内容塞进合适的位置&quot;。</p>
<p>这个独特约束有一个工程含义：<strong>记忆系统的核心 API 不是 <code>get(key)</code> 和 <code>put(key, value)</code>，而是 <code>assemble_context(state, budget) → messages</code></strong>。前者关注存储，后者关注组装——后者才是 Agent 记忆系统的难点。</p>
<hr>
<h2>2. 四层记忆：从对话到知识</h2>
<p>人脑的记忆分层（Atkinson-Shiffrin 模型）对 Agent 设计有强指导意义——不是所有信息都要记，记的方式和位置应该不同。</p>

<p>四层的边界：</p>
<ul>
<li><strong>L1 vs L2</strong>：L1 是&quot;说了什么&quot;，L2 是&quot;正在做什么&quot;。L1 是消息序列，L2 是结构化状态。一个 10 步任务的中间结果塞 L1 会撑爆 context，塞 L2 才是正确做法。</li>

</ul>
<p>四层组合起来注入 LLM 的 context window，形成一次推理需要的全部信息。</p>

<h2>3. 读：怎么组装上下文</h2>
<p>每次 LLM 调用前，从四层里取出相关内容拼成 prompt。这是记忆系统最高频的操作，也是最容易出 bug 的环节。</p>
<h3>上下文组装的伪代码</h3>
<pre><code class="language-python">def assemble_context(
    user_input: str,
    state: AgentState,
    budget: int = 8000,  # token 预算
) -&gt; list[Message]:
    &quot;&quot;&quot;从四层记忆里拼出一次 LLM 调用的 messages&quot;&quot;&quot;

    # 1. System Prompt：固定预算，永远保留
    system = render_system_prompt(state.agent_config)
    system_tokens = count(system)

    # 2. 检索 L3 + L4：按当前 query 找相关经验和知识
    retrieved = retrieve_relevant(
        query=user_input,
        sources=[&quot;L3_episodes&quot;, &quot;L4_docs&quot;],
        top_k=10,
        filter_by_permission=state.user.permissions,
    )
    reranked = rerank(query=user_input, candidates=retrieved, top_k=5)
    retrieval_block = format_retrieval(reranked)

    # 3. L2 任务状态：序列化为简洁文本，不要 dump 整个 JSON
    task_block = format_task_state(state.l2_task)

    # 4. L1 对话历史：压缩到剩余预算
    fixed_tokens = system_tokens + count(retrieval_block) + count(task_block)
    output_reserve = 1500          # 给 LLM 输出留空间
    history_budget = budget - fixed_tokens - count(user_input) - output_reserve

    history = compress_history(state.l1_messages, history_budget)

    return [
        system_msg(system),
        *history,
        context_msg(retrieval_block + task_block),  # 检索结果作为系统级注入
        user_msg(user_input),
    ]
</code></pre>
<p>这段伪代码里有四个值得注意的设计：</p>
<p><strong>System Prompt 永远不裁</strong>。一旦丢失安全规则、工具描述，Agent 行为不可控。</p>
<p><strong>检索结果走系统级注入，而不是 user message</strong>。模型对&quot;system 给的事实&quot;比&quot;user 说的事实&quot;信任度更高。把 RAG 结果伪装成用户说的话是反模式。</p>
<p><strong>L2 任务状态要做摘要</strong>。直接 <code>json.dumps(task_state)</code> 会塞进大量字段 ID、时间戳，LLM 看不懂还浪费 token。要写一个<code>format_task_state</code>把状态翻译成自然语言：&quot;你正在执行任务 X，已完成步骤 1-3，当前在步骤 4，遇到的阻碍是 Y&quot;。</p>
<p><strong>L1 压缩在最后才做</strong>。先把固定预算扣掉，剩下多少才轮到 L1。L1 是&quot;可压缩资源&quot;，其他都是&quot;不可压缩资源&quot;。</p>

<h2>4. 写：什么值得记住</h2>
<p>不是所有交互都值得沉淀。<strong>过度记忆比不记忆更糟</strong>——记忆库噪声多了，检索质量下降，所有依赖记忆的决策都被污染。</p>
<p>写入规则按层不同：</p>

<h3>L4 的写入：事件驱动的&quot;隐形知识库&quot;</h3>
<p>L4 的写入策略和 L1/L2/L3 有本质区别——它不是 Agent 自己触发，而是<strong>文档源的变更触发</strong>。一个跑通的 L4 流水线大致这样：</p>
<pre><code class="language-python">@on_event(&quot;confluence.page.updated&quot;)
def reindex_confluence_page(event):
    &quot;&quot;&quot;Confluence 页面更新 → 增量重建对应 chunk&quot;&quot;&quot;
    doc = fetch_doc(event.page_id)
    # 旧 chunk 标记为 tombstone（不立即删，等查询验证）
    mark_chunks_as_stale(doc_id=event.page_id, version=event.old_version)
    # 解析 + 切块 + embedding + 入库
    chunks = chunking_pipeline(doc)
    embeddings = embed_batch([c.text for c in chunks])
    write_chunks_atomic(chunks, embeddings, doc_id=event.page_id, version=event.new_version)
    # 旧 chunk 被新 chunk 完全覆盖后才物理删除
    gc_stale_chunks(doc_id=event.page_id, older_than=event.new_version)

@on_event(&quot;github.pr.merged&quot;)
def extract_change_log(event):
    &quot;&quot;&quot;PR 合入 → 提取变更说明作为新知识入 L4&quot;&quot;&quot;
    summary = llm.summarize(event.pr_description + event.commits)
    add_to_l4(text=summary, source=event.pr_url, kind=&quot;change_log&quot;)

@on_event(&quot;jira.issue.closed&quot;)
def harvest_faq(event):
    &quot;&quot;&quot;工单关闭 → 抽取 question/answer 作为 FAQ 入 L4&quot;&quot;&quot;
    if event.resolution_quality &gt;= &quot;good&quot;:
        add_to_l4(text=f&quot;Q: {event.summary}\nA: {event.resolution}&quot;, source=event.issue_url)
</code></pre>
<p><strong>事件驱动是核心设计</strong>——靠员工&quot;主动上传文档维护知识库&quot;在企业里几乎注定失败（这是后面 §13 会再点的事实）。把 L4 写入挂到文档真实变更的事件上，知识库才能跟上业务。</p>
<p>L4 写入还有几个易踩的工程坑：<strong>版本切换不能阻塞查询</strong>（用 tombstone + 原子替换而非&quot;先删后建&quot;）、<strong>Embedding 模型升级要全量重建</strong>（旧向量和新查询不在同一空间）、<strong>删除的内容要从向量库 + 缓存 + 搜索引擎多处真正删掉</strong>（否则&quot;已删除文档&quot;仍能被检索到，造成合规风险）。</p>
<h3>L3 写入的决策伪代码</h3>
<p>L3 是最微妙的——写多了污染、写少了浪费。一个工程上跑得通的判定逻辑：</p>
<pre><code class="language-python">def should_write_to_l3(event: AgentEvent) -&gt; tuple[bool, float, str]:
    &quot;&quot;&quot;判断一次 Agent 事件是否值得写入 L3，返回 (是否写, 重要性, 类型)&quot;&quot;&quot;

    # 显式信号：高优先级
    if event.user_feedback == &quot;positive&quot; and event.task_completed:
        return True, 0.9, &quot;success_pattern&quot;
    if event.user_feedback == &quot;negative&quot;:
        return True, 0.95, &quot;failure_lesson&quot;
    if event.kind == &quot;preference_change&quot;:
        return True, 0.85, &quot;preference&quot;

    # 隐式信号：需要 LLM 判断
    if event.kind == &quot;tool_usage&quot; and event.is_first_time:
        # 第一次成功用某个工具的&quot;方法论&quot;值得记
        return True, 0.6, &quot;tool_usage&quot;

    # 失败信号：失败本身就有教学价值
    if event.task_failed and event.has_clear_root_cause:
        return True, 0.8, &quot;failure_lesson&quot;

    # 单次具体事实：不写
    return False, 0.0, &quot;&quot;
</code></pre>
<p>L3 的高门槛意味着情景记忆库不会无限膨胀——一个跑了半年的 Agent，L3 通常只有几千到几万条，远比 L1 的总消息数少。</p>
<h3>L3 写入的反模式</h3>
<p><strong>把整段对话内容当 L3 写</strong> — 这是 L1 的工作。L3 应该写&quot;从这次对话学到的可迁移规律&quot;。</p>
<p><strong>写得太具体</strong> — &quot;用户问了订单 #12345 的状态&quot; 不该进 L3。&quot;用户偏好按订单状态分组展示&quot; 才进 L3。</p>
<p><strong>没有去重</strong> — 同一条经验如果被反复触发写入，会让该规律被检索得过频。写入前要做语义去重——查一下 L3 里有没有相似的记录，有就更新 access_count 和置信度而不是新增。</p>
<pre><code class="language-python">def write_l3(episode: L3_Episode):
    # 语义去重：检查是否已有近似经验
    existing = vector_search(episode.embedding, top_k=3, threshold=0.85)
    if existing:
        # 合并而不是新增——加强已有经验
        existing[0].importance = min(1.0, existing[0].importance + 0.05)
        existing[0].access_count += 1
        existing[0].last_accessed = now()
        update_l3(existing[0])
        return
    # 新经验
    insert_l3(episode)
</code></pre>

<h2>5. 删：遗忘也是必要的</h2>
<p>没有遗忘机制的记忆库最终会被噪声淹没。三种遗忘策略可组合：</p>
<table>
<thead>
<tr>
<th>策略</th>
<th>触发</th>
<th>删除什么</th>
</tr>
</thead>
<tbody><tr>
<td>时间衰减</td>
<td>超过 N 天</td>
<td>重要性 &lt; 0.7 且最后访问 &gt; 30 天前的记录</td>
</tr>
<tr>
<td>容量驱逐</td>
<td>总条数超上限</td>
<td>按&quot;重要性 × 时间新鲜度 × 访问频率&quot;综合排序，删最低分</td>
</tr>
<tr>
<td>显式遗忘</td>
<td>用户要求 / 隐私合规</td>
<td>指定的记录</td>
</tr>
</tbody></table>
<h3>容量驱逐的评分公式</h3>
<pre><code class="language-python">def memory_score(record: L3_Episode, now_ts: int) -&gt; float:
    &quot;&quot;&quot;综合分数越低越优先删除&quot;&quot;&quot;
    age_days = (now_ts - record.created_at) / 86400
    recency_days = (now_ts - record.last_accessed) / 86400

    # 时间新鲜度：用 exp 衰减
    freshness = math.exp(-recency_days / 30)  # 30 天半衰期

    # 访问频率：log 防止过度奖励高频项
    frequency = math.log1p(record.access_count)

    return record.importance * freshness * frequency

def evict_if_needed(capacity: int = 100_000):
    total = count_l3_records()
    if total &lt;= capacity:
        return
    # 按分数升序排，删最低的 N 条
    to_delete = top_k_lowest(score_fn=memory_score, k=total - capacity)
    soft_delete(to_delete, reason=&quot;capacity_eviction&quot;)
</code></pre>
<p><strong>软删除而非硬删除</strong>——支持回滚，避免误删。每次遗忘要记录原因和时间戳，便于审计。</p>
<h3>遗忘的反直觉点</h3>
<p>很多 Agent 团队不做遗忘，因为&quot;删信息感觉是损失&quot;。但实际上：<strong>L3 的检索质量随条数 N 不是线性增长，是先升后降</strong>——少了无内容可检索、多了语义相似的记录互相挤压排名、查准率下降。一个 50 万条的 L3 库未必比 5 万条的更好用，因为信噪比下降得比检索召回率上升得快。所以遗忘机制不是&quot;为节省存储&quot;，是&quot;为维持检索质量&quot;。</p>
<hr>
<h2>6. L4 语义记忆的工程实现：RAG</h2>
<p>RAG（Retrieval-Augmented Generation）是 L4 的具体实现。但 RAG 不是&quot;搜索+拼接&quot;——这种理解会让系统质量天花板极低。</p>
<p><strong>80% 的 RAG 质量问题在检索侧，不在生成侧</strong>。换更贵的模型不如把检索做好。</p>
<p>这个判断有一个朴素的原因：<strong>LLM 没拿到的信息它就答不出来，拿到了错的信息它就编出错的答案</strong>。检索阶段的失败（漏检、错检）会直接体现为生成阶段的失败（不知道、瞎编）。反过来，生成阶段的问题（输出格式不对、引用不准）通常是 prompt 工程问题，不是模型能力问题。<strong>所以 RAG 项目调优的本质是检索调优</strong>——绝大部分 PoC 阶段表现差的 RAG 项目，问题都在 chunking、embedding、retrieval、rerank 这条链路上。</p>
<p>RAG 的完整流水线分两条链：</p>
<table>
<thead>
<tr>
<th>链</th>
<th>阶段</th>
<th>关键决策</th>
</tr>
</thead>

<td>文档解析 → Chunking → Embedding → 入向量库</td>
<td>怎么切、用什么 embedding、怎么分区</td>

</tr>
</tbody></table>

<h2>7. Chunking：切得不好，后面全废</h2>
<p>Chunking 是最被低估、对质量影响最大的环节。切分策略直接决定：</p>
<ul>
<li>检索时能否命中相关内容</li>
<li>命中的内容是否包含足够上下文</li>
<li>LLM 拿到的信息有多少噪声</li>
</ul>
<table>
<thead>
<tr>
<th>策略</th>
<th>思路</th>
<th>问题</th>
</tr>
</thead>
<tbody><tr>
<td>固定长度</td>
<td>按字符数等间隔切</td>
<td>完全不考虑语义边界，会把句子切成两半</td>
</tr>
<tr>
<td>语义切分</td>
<td>按段落、标题、代码块</td>
<td>块大小不均，依赖文档格式规范</td>
</tr>
<tr>
<td>递归切分</td>
<td>先按大边界（\n\n），切不动再用小边界（\n、.）</td>
<td>平衡了语义和大小，是 LangChain <code>RecursiveCharacterTextSplitter</code> 的核心思路</td>
</tr>
</tbody></table>
<h3>递归切分的伪代码</h3>
<pre><code class="language-python">def recursive_split(text: str, max_size: int = 512, overlap: int = 50) -&gt; list[str]:
    &quot;&quot;&quot;递归切分：先按大边界切，块仍然过大就用更小的边界继续切&quot;&quot;&quot;
    separators = [&quot;\n\n&quot;, &quot;\n&quot;, &quot;。&quot;, &quot;！&quot;, &quot;？&quot;, &quot;. &quot;, &quot;; &quot;, &quot; &quot;]

    if len(text) &lt;= max_size:
        return [text]

    for sep in separators:
        if sep in text:
            parts = text.split(sep)
            chunks, buf = [], &quot;&quot;
            for p in parts:
                p = p + sep
                if len(buf) + len(p) &lt;= max_size:
                    buf += p
                else:
                    if buf:
                        chunks.append(buf)
                    if len(p) &gt; max_size:
                        # 单个 part 还太大，递归切
                        chunks.extend(recursive_split(p, max_size, overlap))
                        buf = &quot;&quot;
                    else:
                        buf = p
            if buf:
                chunks.append(buf)
            return add_overlap(chunks, overlap)

    # 所有分隔符都没用，只能硬切
    return [text[i : i + max_size] for i in range(0, len(text), max_size - overlap)]
</code></pre>
<h3>Parent-Document Retrieval</h3>
<p>当前公认最有效的落地技巧：<strong>用小 chunk 做检索，用大 chunk 给模型</strong>。小 chunk（<del>200 tokens）语义集中、检索准；命中后返回 parent chunk（</del>2000 tokens）让模型看到完整上下文。解决了&quot;切大不准、切小不够&quot;的两难。</p>
<pre><code class="language-python">def index_with_parents(doc: str):
    &quot;&quot;&quot;双层 chunking：小 chunk 用于检索，大 chunk 用于喂模型&quot;&quot;&quot;
    parents = recursive_split(doc, max_size=2000, overlap=200)
    for parent in parents:
        parent_id = save_chunk(parent, role=&quot;parent&quot;)
        children = recursive_split(parent, max_size=200, overlap=20)
        for child in children:
            save_chunk(child, role=&quot;child&quot;, parent_id=parent_id, embedding=embed(child))

def retrieve_with_parents(query: str, top_k: int = 5):
    &quot;&quot;&quot;检索时取 child，返回 parent&quot;&quot;&quot;
    child_hits = vector_search(query, role=&quot;child&quot;, top_k=top_k * 3)
    parent_ids = unique([c.parent_id for c in child_hits])[:top_k]
    return load_chunks(parent_ids)
</code></pre>
<p><strong>Chunk overlap</strong>：相邻 chunk 重叠 10-20% 防止边界信息丢失。一个 512 token 的 chunk 配 50-100 token overlap 是合理起点。</p>

<h3>混合检索的 RRF 伪代码</h3>
<p><strong>生产系统默认 Hybrid</strong>。融合用 RRF（Reciprocal Rank Fusion）——不关心绝对分数，只关心排名：</p>
<pre><code class="language-python">def hybrid_retrieve(query: str, top_k: int = 20, k: int = 60) -&gt; list[Chunk]:
    &quot;&quot;&quot;BM25 + Vector，用 RRF 融合排名&quot;&quot;&quot;
    bm25_hits = bm25_search(query, top_k=top_k * 2)       # [(chunk, rank), ...]
    vector_hits = vector_search(query, top_k=top_k * 2)

    scores = defaultdict(float)
    for hits in (bm25_hits, vector_hits):
        for rank, hit in enumerate(hits, start=1):
            # RRF 公式：1 / (k + rank)
            scores[hit.id] += 1 / (k + rank)

    # 按融合分数排序，取 top_k
    sorted_ids = sorted(scores.keys(), key=lambda x: -scores[x])[:top_k]
    return load_chunks(sorted_ids)
</code></pre>
<p>RRF 的好处是不需要在 BM25 分数（一般 5-30）和 Cosine 分数（0-1）之间做归一化——它只看排名。k 常取 60，是经验值。</p>
<p>只有非常确定场景只需要语义搜索（纯自然语言文档、无 ID）才用纯 Vector。</p>

<h2>9. Reranking：从召回到精排</h2>
<p>初步检索的目标是<strong>高召回率</strong>——把相关文档都捞出来，但排在前面的不一定最相关。Reranking 用更强的模型对 Top-50 做精排选出 Top-5。</p>

<h3>两阶段检索的伪代码</h3>
<pre><code class="language-python">def retrieve_with_rerank(
    query: str,
    initial_k: int = 50,
    final_k: int = 5,
) -&gt; list[Chunk]:
    &quot;&quot;&quot;两阶段检索：粗排 + 精排&quot;&quot;&quot;
    # 第一阶段：Hybrid 召回 50 个候选
    candidates = hybrid_retrieve(query, top_k=initial_k)

    # 第二阶段：Cross-encoder Reranker 精排
    pairs = [(query, c.text) for c in candidates]
    rerank_scores = cross_encoder.score(pairs)  # 例如 BGE-Reranker-v2

    # 按 rerank 分数取 top final_k
    ranked = sorted(zip(candidates, rerank_scores), key=lambda x: -x[1])
    return [c for c, _ in ranked[:final_k]]
</code></pre>
<p>Reranker 选型：精度优先用 Cohere Rerank 或 BGE-Reranker-v2，成本优先用小参数量 Cross-encoder。</p>
<h3>Reranker 的成本考虑</h3>
<p>Reranker 每次要算 N 个 (query, doc) 对——50 个候选就是 50 次推理。Cross-encoder 比 Bi-encoder 慢 10-100 倍，所以<strong>只在小批量上用</strong>。直接对全库做 reranking 是噩梦：100 万文档 = 100 万次推理。两阶段架构（Bi-encoder 粗排 → Cross-encoder 精排）正是为了规避这个问题。</p>

<h2>10. Context Packing：信息怎么送达 LLM</h2>
<p>检索+重排后拿到 Top-K chunks，接下来是怎么组织送进 prompt。这一步直接决定 LLM 信息利用率。</p>
<h3>Lost in the Middle 现象</h3>
<p>Stanford 2023 年的论文《Lost in the Middle: How Language Models Use Long Contexts》（Liu et al., arXiv:2307.03172）通过 NaturalQuestions 数据集系统测量了多个商业 LLM——在 20 个文档的 context 中，把&quot;含答案的文档&quot;放在不同位置时模型回答准确率呈 U 型分布：放在开头约 75%、放在结尾约 65%、放在中间最低约 50%。这意味着：<strong>最相关的文档应该放开头或结尾，不放中间</strong>。这个现象后续在 GPT-4、Claude、Gemini 各代模型上都被独立复现，是 RAG 工程的稳定假设而非短期 bug。</p>
<p><img src="/images/blog/agentic/lost-in-middle.svg" alt="Lost in the Middle"></p>
<p>两种排布策略：</p>
<table>
<thead>
<tr>
<th>策略</th>
<th>排序</th>
<th>适合</th>
</tr>
</thead>
<tbody><tr>
<td>Relevance-first</td>
<td>按相关性降序排列</td>
<td>上下文较短（&lt; 5 chunks）</td>
</tr>
<tr>
<td>Edges-first</td>
<td>最相关放头尾，次相关放中间</td>
<td>上下文较长（≥ 5 chunks）</td>
</tr>
</tbody></table>
<h3>Edges-first 的伪代码</h3>
<pre><code class="language-python">def edges_first_arrange(chunks: list[Chunk]) -&gt; list[Chunk]:
    &quot;&quot;&quot;最相关放两端，次相关放中间&quot;&quot;&quot;
    # 输入按相关性已排序：chunks[0] 最相关
    n = len(chunks)
    result = [None] * n
    left, right = 0, n - 1
    for i, chunk in enumerate(chunks):
        if i % 2 == 0:
            result[left] = chunk
            left += 1
        else:
            result[right] = chunk
            right -= 1
    return result
# 5 个 chunks 按相关性 [1,2,3,4,5] → 排列为 [1, 3, 5, 4, 2]
</code></pre>
<h3>Token 预算的硬约束</h3>
<pre><code class="language-python">def pack_context(chunks: list[Chunk], budget: int = 3000) -&gt; str:
    &quot;&quot;&quot;按 token 预算打包检索结果&quot;&quot;&quot;
    packed = []
    used = 0
    for chunk in chunks:
        chunk_tokens = count(chunk.text)
        if used + chunk_tokens &gt; budget:
            break  # 宁可少放，不要截断
        packed.append(chunk)
        used += chunk_tokens

    arranged = edges_first_arrange(packed)
    return format_with_citations(arranged)

def format_with_citations(chunks: list[Chunk]) -&gt; str:
    &quot;&quot;&quot;给每个 chunk 加来源标识，方便 LLM 生成 citation&quot;&quot;&quot;
    return &quot;\n\n&quot;.join([
        f&quot;[Source {i+1}: {c.metadata.title}#{c.metadata.section_path[-1]}]\n{c.text}&quot;
        for i, c in enumerate(chunks)
    ])
</code></pre>
<p>宁可少放几个 chunk、每个完整，也<strong>不要截断 chunk 送进去</strong>——被截断的信息比没有更糟。给每个 chunk 附来源标识，方便 LLM 生成 citation。</p>

<h2>11. RAG 评估：检索侧的特有指标</h2>
<p><strong>不可度量则不可改进</strong>。这里只讲 RAG 流水线特有的检索侧指标——Agent 整体的评估体系（任务完成度、轨迹质量、LLM-as-Judge 的偏差与校准、Quality Gate）是独立的工程话题，本节不展开。</p>

<h3>Faithfulness 的 LLM-as-Judge 模板</h3>
<p>Faithfulness 是最关键的——检索回来的内容对的，但 LLM 基于这些内容编造了一个看似合理实际错误的回答，比没找到信息更危险。Faithfulness 评估的 prompt 骨架：</p>
<pre><code class="language-text">你是一个事实核查员。判断下面的&quot;回答&quot;是否完全基于&quot;上下文&quot;。

规则：
1. 拆解回答为原子陈述（atomic claim），逐条核查
2. 每条 claim 必须在 context 中有明确依据
3. 推论、扩展、举例如果 context 没有支持，也算不忠实
4. 输出每条 claim 的核查结果

上下文：
{retrieved_context}

回答：
{generated_answer}

输出 JSON：
{
  &quot;claims&quot;: [
    {&quot;claim&quot;: &quot;...&quot;, &quot;supported&quot;: true|false, &quot;evidence&quot;: &quot;context 中的原文&quot; }
  ],
  &quot;faithfulness_score&quot;: &lt;supported claims 数 / 总 claim 数&gt;,
  &quot;unsupported_claims&quot;: [&quot;...&quot;]
}
</code></pre>
<h3>评估的工程流程</h3>
<pre><code class="language-python">def run_rag_eval(test_set: list[TestCase]) -&gt; EvalReport:
    &quot;&quot;&quot;跑一遍评估集，比较当前 pipeline 与基线&quot;&quot;&quot;
    results = []
    for case in test_set:
        retrieved = retrieve_with_rerank(case.question)
        answer = generate(case.question, retrieved)

        results.append({
            &quot;case_id&quot;: case.id,
            &quot;recall_at_5&quot;: metric_recall(retrieved, case.gold_docs, k=5),
            &quot;mrr&quot;: metric_mrr(retrieved, case.gold_docs),
            &quot;faithfulness&quot;: llm_judge_faithfulness(answer, retrieved),
            &quot;answer_relevancy&quot;: llm_judge_relevancy(answer, case.question),
        })
    return aggregate(results)
</code></pre>
<p>实操：建 50-100 条的评估集（question + ground truth），每次 Pipeline 变更（换 Embedding、调 Chunking、加 Reranker）跑一轮，关注指标变化方向而非绝对数值。<strong>绝对数值不重要，重要的是变化方向</strong>——一次实验如果 Recall 升 5%、Faithfulness 持平、Answer Relevancy 升 3%，就值得上线；如果 Recall 升但 Faithfulness 下降，说明召回了更多噪声、得不偿失。</p>

<h2>12. RAG 与 Long Context 的关系</h2>
<p>2024-2025 年最常被问的：Gemini 支持 1M+ tokens、Claude 200K、GPT-4o 128K，还需要 RAG 吗？</p>
<p>需要，但定位变了。</p>
<table>
<thead>
<tr>
<th>维度</th>
<th>Long Context</th>
<th>RAG</th>
</tr>
</thead>
<tbody><tr>
<td>数据规模</td>
<td>几十万字以内</td>
<td>几百万到亿级字</td>
</tr>
<tr>
<td>成本</td>
<td>按 token 计费，塞越多越贵</td>
<td>检索成本低，只送相关片段</td>
</tr>
<tr>
<td>精度</td>
<td>大海捞针仍有盲区（中间位置）</td>
<td>检索质量可控可调</td>
</tr>
<tr>
<td>权限</td>
<td>全塞进去无法做细粒度权限</td>
<td>可按用户/角色过滤</td>
</tr>
<tr>
<td>更新</td>
<td>每次请求都要重新塞</td>
<td>知识库持久化，增量更新</td>
</tr>
</tbody></table>
<p><strong>互补不是替代</strong>：Long Context 适合&quot;少量文档的深度理解&quot;，RAG 适合&quot;海量知识的精准检索&quot;。实际工程中两者常组合——RAG 从百万文档中筛 Top-K 片段，Long Context 让模型深度理解这些片段。</p>
<p>一个常见的误判是&quot;Long Context 让 RAG 过时了&quot;。这个说法表面成立——既然能塞 1M token 进去为什么还要检索？深入看会发现四个事实让它站不住：</p>
<p><strong>成本随长度线性增长</strong>。Gemini 1.5 Pro 输入定价（截至 2026 年初）约 $1.25/1M token（≤128K）、$2.50/1M token（&gt;128K），1M token 的单次调用约 $2.50。一个日均 10 万次的请求路径上塞满 1M token，每月几乎是 $750K。RAG 把每次请求的输入压到 5-20K，成本是 Long Context 路径的 1-3%。</p>
<p><strong>大海捞针测试在公开 benchmark 上效果很好，但在企业实际文档上表现远不如想象</strong>。Needle-in-Haystack 是把单条已知信息插入随机文本里——这种&quot;信号 vs 噪声完全独立&quot;的设置在真实企业文档中根本不存在。真实文档里充满表格、引用、专业术语、版本号、近似但不相同的语句——这些都是&quot;非噪声&quot;但也&quot;非答案&quot;，会把 attention 真正分散掉。Anthropic 在 2024 年公开过 Claude 在&quot;修改版 needle-in-haystack&quot;（信号被语义近似项干扰）上的下降——准确率在 64K+ 区段下降 20-40 个百分点。</p>
<p><strong>Attention 在长序列上有效性递减</strong>。即使模型能&quot;看到&quot;所有内容，也不代表能&quot;用好&quot;所有内容。这跟 Lost in the Middle 是同一回事——中间区域的有效利用率显著低。1M context 里如果关键事实在 50 万 token 位置，被有效使用的概率显著低于在前 10 万 token 内。</p>
<p><strong>权限隔离和增量更新在 Long Context 下都做不了</strong>。你不可能为每个用户单独维护一个 100 万 token 的 prompt——租户隔离需要按角色过滤的能力，这只能在检索层做。文档更新也不可能让&quot;每次请求都重新塞 1M token&quot;——必须有增量索引机制，这也是 RAG 的能力。</p>
<p><strong>未来的趋势是更长的 context 加上更聪明的检索，而不是 context 取代检索</strong>。Long Context 让&quot;少量但要深读&quot;的场景变好，但&quot;海量、可过滤、增量更新、按权限隔离&quot;的场景永远是 RAG 的主场。</p>

<h2>14. 记忆是结构问题，不是容量问题</h2>
<p>记忆系统的核心难题从来不是&quot;能记多少&quot;——一个 8GB 的向量数据库存几百万条事实毫无压力。难题是<strong>结构</strong>：什么写在哪一层、什么时候读、什么时候删、不同层之间怎么协同。L1 对话缓冲解决&quot;刚说过什么&quot;、L2 工作记忆解决&quot;这个任务我做到哪了&quot;、L3 情景记忆解决&quot;以前类似情况怎么处理的&quot;、L4 语义记忆解决&quot;领域知识是什么&quot;——四层各自的访问频率、更新频率、容量需求都不同，混在一起就是灾难。</p>
<p>语义记忆的工程形态是 RAG，但 RAG 不是&quot;搜索+拼接&quot;那么简单——它是 Chunking、Embedding、混合检索、Reranking、Context Packing 的完整流水线，每一环都有自己的失败模式。质量瓶颈 80% 在检索侧而不是生成侧——这就是为什么 RAG 项目调优的本质是检索调优，换大模型基本无效。</p>
<p>Long Context 不会让 RAG 过时——它会让检索从&quot;必须做&quot;变成&quot;几个场景下也许可以不做&quot;。但只要你还需要权限隔离、低延迟、按场景动态加载知识，RAG 的位置就稳得很。</p>
1e:Tbcb2,<blockquote>
<p>微服务架构的核心难题不是技术选型，而是<strong>如何找到正确的服务边界</strong>。拆分得太粗，和单体无异；拆分得太细，分布式的复杂性会吞噬所有收益。领域驱动设计（DDD）提供了一套从问题空间到解决方案空间的完整方法论——战略设计回答&quot;边界在哪里&quot;，战术设计回答&quot;边界内怎么建&quot;。本文以电商领域为主线，从子域划分、事件风暴、上下文映射，一路推进到聚合设计、Entity、Value Object、Domain Service 等战术构件的代码落地，并覆盖 Outbox、Saga、CQRS 等分布式一致性方案。</p>
</blockquote>
<h2>微服务的本质：不是&quot;小&quot;，而是&quot;界限清晰&quot;</h2>
<p>微服务中的&quot;微&quot;容易让人把关注点放在服务的规模上，但规模从来不是核心标准。Adrian Cockcroft 对微服务有一个精炼的定义：</p>
<blockquote>
<p>&quot;面向服务的架构由具有<strong>界限上下文</strong>、<strong>松散耦合</strong>的元素组成。&quot;</p>
</blockquote>
<p>一个真正的微服务架构应当具备以下特征：</p>
<table>

<tr>

<th>说明</th>
</tr>
</thead>
<tbody><tr>
<td>业务边界清晰</td>
<td>服务以业务上下文为中心，而非技术抽象</td>
</tr>
<tr>
<td>实现细节隐藏</td>
<td>通过意图接口暴露功能，不泄露内部实现</td>
</tr>

<p>归纳起来：<strong>松散耦合的面向服务架构，每个服务封装在定义良好的界限上下文中，支持快速、频繁且可靠的交付。</strong></p>
<p>微服务的强大之处在于：<strong>边界内建立高内聚，边界外建立低耦合</strong>——倾向于一起改变的事物应该放在一起。但说起来容易做起来难，业务在不断发展，设想也随之改变。因此，<strong>重构能力</strong>是设计系统时必须考虑的关键问题。</p>
<h2>DDD 战略设计：从问题空间到解决方案空间</h2>
<p>领域驱动设计（Domain-Driven Design）因 Eric Evans 的同名著作而闻名，它是一组思想、原则和模式，帮助我们基于业务领域的底层模型来设计软件系统。战略设计是 DDD 的宏观层面，关注的是如何划分系统边界、建立团队间的协作契约。</p>
<h3>基本术语</h3>

<h3>子域分类：核心域、支撑域与通用域</h3>
<p>并非所有子域对业务的价值相同。Eric Evans 将子域分为三类，这个分类直接决定了资源投入的优先级和技术决策：</p>
<table>
<thead>
<tr>
<th>类型</th>
<th>特征</th>
<th>投入策略</th>
<th>电商示例</th>

<tbody><tr>
<td><strong>核心域（Core Domain）</strong></td>
<td>业务的差异化竞争力所在，是组织最独特的能力</td>
<td>投入最优秀的团队，自研，持续迭代</td>
<td>推荐算法、定价策略、供应链优化</td>
</tr>
<tr>

<h3>界限上下文：同一个词，不同的含义</h3>
<p>以电商系统中的 <strong>&quot;Item&quot;（商品）</strong> 为例，它在不同的上下文中有着截然不同的含义：</p>

<th>关注的属性</th>

<tbody><tr>
<td><strong>Catalog（目录）</strong></td>
<td>可出售的产品</td>
<td>名称、描述、价格、图片、分类</td>

<p><strong>子域 vs 界限上下文</strong>：子域属于<strong>问题空间</strong>（业务如何看待问题），界限上下文属于<strong>解决方案空间</strong>（如何实现问题的解决方案）。理论上一个子域可以有多个界限上下文，但我们努力做到每个子域只有一个。</p>
</blockquote>
<h2>事件风暴：协作式的边界发现</h2>
<p>有了子域和界限上下文的概念，下一个问题是：<strong>具体的边界怎么找出来？</strong> 事件风暴（Event Storming）是 Alberto Brandolini 提出的一种轻量级的协作建模技术，它是将业务知识转化为聚合边界和上下文映射的实操工具。</p>
<h3>什么是事件风暴？</h3>
<p>简单来说，事件风暴是团队在一起进行的头脑风暴，目标是识别系统中发生的各种<strong>领域事件</strong>和<strong>业务流程</strong>。所有相关团队在同一个房间（物理或虚拟），在白板上用不同颜色的便利贴标记事件、命令、聚合和策略，识别重叠概念、模糊的领域语言和冲突的业务流程，最后对相关模型进行分组，发现聚合边界。</p>
<h3>便利贴颜色约定</h3>
<table>
<thead>
<tr>
<th>颜色</th>
<th>含义</th>
<th>示例</th>
</tr>
</thead>
<tbody><tr>
<td>橙色</td>

<h3>从事件风暴到聚合边界</h3>
<p>事件风暴的关键产出不是事件清单本身，而是通过事件的聚类发现<strong>聚合边界</strong>。具体做法是：先把所有领域事件铺到时间线上，然后把频繁一起出现的事件归拢到同一个聚合，最后观察哪些聚合之间的交互是事件驱动的（松耦合），哪些需要同步调用（紧耦合）。紧耦合的聚合倾向于放在同一个服务内，松耦合的聚合适合拆分为不同服务。</p>
<p>以电商下单场景为例：把&quot;商品已添加到购物车&quot;、&quot;购物车已更新&quot;归入购物车聚合；把&quot;订单已创建&quot;、&quot;订单行已添加&quot;、&quot;订单已提交&quot;归入订单聚合；把&quot;支付已授权&quot;、&quot;支付已完成&quot;、&quot;支付已退款&quot;归入支付聚合。然后观察到&quot;订单已提交&quot;之后触发&quot;支付已授权&quot;是跨聚合的事件流——这两个聚合之间是松耦合的，适合拆分为不同服务。而&quot;订单已创建&quot;和&quot;订单行已添加&quot;总是在同一个事务中发生——它们属于同一个聚合。</p>
<p>一次成功的事件风暴通常会产出：重新定义的聚合列表（这些可能成为新的微服务）、聚合之间的事件流（上下文映射的输入）、热点问题（需要进一步讨论的边界争议）、以及团队对统一语言的共同理解。<strong>事件风暴的产出直接喂入下一步的上下文映射。</strong></p>
<h2>上下文映射：定义服务间的协作契约</h2>
<p>事件风暴帮我们发现了聚合和它们之间的事件流，上下文映射（Context Mapping）则在此基础上进一步定义：<strong>这些上下文之间应该用什么模式协作？</strong> DDD 定义了一组标准化的映射模式，每种模式反映了不同的团队关系和技术集成方式。</p>

<h3>防腐层：隔离外部模型污染</h3>
<p>在实际项目中，我们不可避免地要对接第三方系统——支付网关、物流平台、ERP 系统。这些系统的数据模型和你的领域模型往往差异巨大。如果直接在领域代码中使用第三方的数据结构，外部的任何变更都会直接穿透到核心业务逻辑。</p>
<p>防腐层的职责是在两个世界之间做翻译：</p>
<pre><code class="language-java">// 防腐层：将第三方支付网关的响应转换为内部领域模型
public class PaymentGatewayACL {
    private final ExternalPaymentClient client;

    public PaymentResult authorize(PaymentCommand cmd) {
        // 将内部领域命令转为外部 API 请求
        GatewayRequest request = GatewayRequest.builder()
            .merchantId(config.getMerchantId())
            .amount(cmd.getAmount().toCents())
            .currency(cmd.getAmount().getCurrency().name())
            .cardToken(cmd.getTokenizedCard())
            .build();

        // 调用外部服务
        GatewayResponse response = client.charge(request);

        // 将外部响应转为内部领域模型
        return PaymentResult.builder()
            .transactionId(TransactionId.of(response.getTxnRef()))
            .status(mapStatus(response.getResultCode()))
            .authorizedAmount(Money.of(response.getSettledAmount(), cmd.getAmount().getCurrency()))
            .build();
    }

    private PaymentStatus mapStatus(String gatewayCode) {
        return switch (gatewayCode) {
            case &quot;00&quot; -&gt; PaymentStatus.AUTHORIZED;
            case &quot;05&quot; -&gt; PaymentStatus.DECLINED;
            default -&gt; PaymentStatus.FAILED;
        };
    }
}
</code></pre>
<p>ACL 的价值在于：当第三方网关从 v2 升级到 v3、字段名和状态码全部变化时，你只需要修改 ACL 内部的映射逻辑，领域层完全不受影响。</p>

<h3>何时应该合并而非拆分？</h3>
<p>如果发现两个聚合之间需要强 ACID 事务，这是一个强烈的信号——它们可能应该属于同一个聚合，或者至少属于同一个微服务。在拆分之前，事件风暴和上下文映射可以帮助我们及早识别这些依赖关系。</p>
<h2>战术设计：边界内的领域建模</h2>
<p>战略设计确定了服务边界，接下来的问题是：<strong>边界内部怎么建？</strong> 这就是战术设计的领域——一组用于构建丰富领域模型的构件，其中<strong>聚合</strong>是最核心的概念。</p>
<h3>聚合（Aggregate）：数据一致性的原子边界</h3>
<p>聚合是 DDD 中最重要也最容易被误用的概念。它是<strong>数据变更的原子边界</strong>——一次事务只能修改一个聚合。</p>
<blockquote>
<p>聚合是关联对象的集群，被视为数据变更的单元。外部引用仅限于指定聚合的一个成员——<strong>聚合根（Aggregate Root）</strong>。在聚合的边界内需应用一组一致性规则。</p>
</blockquote>
<p>三条铁律：一致性在单个聚合内保证（跨聚合只能最终一致）；只能通过聚合根的已发布接口修改聚合（外部不能绕过聚合根直接操作内部对象）；任何违反这些规则的行为都有让应用退化为大泥球的风险。</p>
<h3>聚合设计原则</h3>
<p>Vaughn Vernon 在《Implementing Domain-Driven Design》中总结了四条聚合设计原则，这四条原则比定义本身更有实操价值：</p>
<p><strong>原则一：在聚合边界内保护业务不变量。</strong> 聚合存在的理由是保护不变量（Invariant）。例如&quot;订单总金额必须等于所有订单行金额之和&quot;这条规则，Order 和 OrderLine 必须在同一个聚合内，因为它们共同维护这个不变量。</p>
<p><strong>原则二：设计小聚合。</strong> 大聚合意味着大事务、大锁、高冲突。一个包含几十个实体的聚合，每次修改都要加载和锁定整个对象图——并发性能会急剧下降。尽量让聚合只包含维护不变量所必需的最小实体集合。</p>
<p><strong>原则三：通过唯一标识引用其他聚合，而非直接对象引用。</strong> 如果 Order 聚合需要关联 Customer，不要在 Order 内部持有 Customer 对象的引用，而是持有 CustomerId。这强制了聚合边界，避免了跨聚合的事务。</p>
<p><strong>原则四：使用最终一致性更新其他聚合。</strong> 当一个聚合的变更需要触发另一个聚合的更新时，通过领域事件实现异步通知，而不是在同一个事务中同时修改两个聚合。</p>
<h3>聚合设计实战：订单聚合</h3>
<pre><code class="language-java">// 聚合根
public class Order {
    private OrderId id;
    private CustomerId customerId;   // 通过 ID 引用，而非持有 Customer 对象
    private OrderStatus status;
    private Money totalAmount;
    private List&lt;OrderLine&gt; lines;   // OrderLine 是聚合内部的实体
    private List&lt;DomainEvent&gt; events = new ArrayList&lt;&gt;();

    // 业务行为封装在聚合根上
    public void addLine(ProductId productId, int quantity, Money unitPrice) {
        if (status != OrderStatus.DRAFT) {
            throw new OrderAlreadySubmittedException(id);
        }
        OrderLine line = new OrderLine(productId, quantity, unitPrice);
        this.lines.add(line);
        recalculateTotal();   // 保护不变量：总金额 = Σ 行金额
    }

    public void submit() {
        if (lines.isEmpty()) {
            throw new EmptyOrderException(id);
        }
        this.status = OrderStatus.SUBMITTED;
        events.add(new OrderSubmitted(id, customerId, totalAmount, lines, Instant.now()));
    }

    public void cancel(String reason) {
        if (!status.isCancellable()) {
            throw new OrderNotCancellableException(id, status);
        }
        this.status = OrderStatus.CANCELLED;
        events.add(new OrderCancelled(
            UUID.randomUUID().toString(),
            id, customerId, totalAmount, reason,
            toCancelledItems(lines), Instant.now(), nextEventVersion()));
    }

    private void recalculateTotal() {
        this.totalAmount = lines.stream()
            .map(OrderLine::lineAmount)
            .reduce(Money.ZERO, Money::add);
    }
}
</code></pre>
<p>这个设计体现了几个关键点：Order 是聚合根，OrderLine 是内部实体，不对外暴露；Customer 通过 CustomerId 引用，不在 Order 聚合内；所有业务行为（addLine、submit、cancel）都封装在聚合根上，由聚合根保护不变量；状态变更产生领域事件，由聚合内部收集，持久化时一并发布。</p>
<p>如何判断一个聚合是不是太大了？有几个信号：事务冲突频繁（多个用户同时操作同一个聚合）、加载一个聚合需要查询多张表和大量数据、一个看似简单的修改要锁定大量对象。反过来，聚合太小也有问题：如果你发现两个聚合之间需要强 ACID 一致性，它们可能应该属于同一个聚合。</p>
<h3>实体（Entity）与值对象（Value Object）</h3>
<p>这是 DDD 战术设计中最基本的二分法：</p>

<thead>

<th>维度</th>

<td>标识</td>
<td>有唯一标识，即使属性完全相同的两个实例也不相等</td>

<p>值对象是被严重低估的战术构件。很多团队习惯性地把所有概念建模为实体，导致领域模型充斥着不必要的 ID 和可变状态。一个好的判断标准：<strong>如果你关心的是&quot;它是哪一个&quot;，用实体；如果你关心的是&quot;它是什么值&quot;，用值对象。</strong></p>
<pre><code class="language-java">// 值对象：Money——不可变、无标识、通过值判断相等
public record Money(BigDecimal amount, Currency currency) {
    public Money {
        if (amount.scale() &gt; currency.getDefaultFractionDigits()) {
            throw new IllegalArgumentException(&quot;Amount scale exceeds currency precision&quot;);
        }
    }

    public Money add(Money other) {
        assertSameCurrency(other);
        return new Money(amount.add(other.amount), currency);
    }

    public Money multiply(int quantity) {
        return new Money(amount.multiply(BigDecimal.valueOf(quantity)), currency);
    }

    public static final Money ZERO = new Money(BigDecimal.ZERO, Currency.getInstance(&quot;CNY&quot;));
}
</code></pre>
<pre><code class="language-java">// 值对象：Address——两个 Address 相等，当且仅当所有字段都相等，不需要 ID
public record Address(String province, String city, String district, String street, String zipCode) {}
</code></pre>
<h3>领域服务（Domain Service）</h3>
<p>当一个业务操作不自然地属于任何一个实体或值对象时，它应该放在领域服务中。领域服务的特点是：无状态、操作涉及多个聚合或外部信息、名称来自统一语言。</p>
<pre><code class="language-java">// 领域服务：定价策略
public class PricingService {
    private final DiscountPolicyRepository discountRepo;

    /**
     * 计算订单的最终价格，涉及多条折扣规则的组合应用。
     * 这个逻辑不属于 Order（Order 不应该知道折扣规则的细节），
     * 也不属于 DiscountPolicy（单条规则不知道其他规则的存在），
     * 因此放在领域服务中。
     */
    public Money calculateFinalPrice(Order order, CustomerId customerId) {
        List&lt;DiscountPolicy&gt; policies = discountRepo.findApplicable(customerId, order.getLines());
        Money baseTotal = order.getTotalAmount();

        for (DiscountPolicy policy : policies) {
            baseTotal = policy.apply(baseTotal, order);
        }
        return baseTotal;
    }
}
</code></pre>
<p>一个常见误区是把所有业务逻辑都放在领域服务中，让实体退化为只有 getter/setter 的数据容器——这就是**贫血领域模型（Anemic Domain Model）**反模式，后面会专门讨论。</p>
<h3>仓储（Repository）与应用服务（Application Service）</h3>
<p>Repository 为聚合提供类集合的持久化接口，屏蔽底层存储细节。设计原则是<strong>一个聚合根对应一个 Repository</strong>，内部实体不需要独立的 Repository。</p>
<pre><code class="language-java">public interface OrderRepository {
    Optional&lt;Order&gt; findById(OrderId id);
    void save(Order order);
    List&lt;Order&gt; findByCustomerId(CustomerId customerId);
}
</code></pre>
<p>应用服务是领域模型的使用者，负责编排用例流程：接收外部请求、加载聚合、调用领域行为、持久化结果、发布事件。它本身<strong>不包含业务逻辑</strong>。</p>
<pre><code class="language-java">@Service
@Transactional
public class OrderApplicationService {
    private final OrderRepository orderRepo;
    private final PricingService pricingService;
    private final DomainEventPublisher eventPublisher;

    public void submitOrder(SubmitOrderCommand cmd) {
        // 1. 加载聚合
        Order order = orderRepo.findById(cmd.getOrderId())
            .orElseThrow(() -&gt; new OrderNotFoundException(cmd.getOrderId()));

        // 2. 调用领域服务（跨聚合逻辑）
        Money finalPrice = pricingService.calculateFinalPrice(order, cmd.getCustomerId());
        order.applyFinalPrice(finalPrice);

        // 3. 调用聚合的业务行为
        order.submit();

        // 4. 持久化
        orderRepo.save(order);

        // 5. 发布领域事件
        order.getDomainEvents().forEach(eventPublisher::publish);
    }
}
</code></pre>
<p>应用服务与领域服务的区别在于：<strong>应用服务编排&quot;做什么&quot;，领域服务封装&quot;怎么做&quot;。</strong> 应用服务知道用例的流程步骤，但不知道业务规则的细节；领域服务知道业务规则，但不知道自己在哪个用例流程中被调用。</p>
<h3>分层架构与六边形架构</h3>
<p>战术设计的构件需要一个合理的代码组织方式。传统的 DDD 分层架构将代码分为四层：</p>
<table>
<thead>
<tr>

<th>职责</th>

</tr>

<td>处理 HTTP/gRPC 请求，参数校验，DTO 转换</td>

<p>这里的关键约束是<strong>依赖方向</strong>：上层可以依赖下层，Domain 层不依赖任何外层。这意味着 Repository 的接口定义在 Domain 层，实现放在 Infrastructure 层——依赖倒置原则的典型应用。</p>
<p>更进一步的做法是<strong>六边形架构（Hexagonal Architecture）</strong>，也叫端口-适配器架构。它的核心思想是：领域模型在最内层，所有外部交互通过端口（Port，即接口）和适配器（Adapter，即实现）进行，无论是 HTTP 请求、数据库访问还是消息队列，对领域层来说都是可替换的适配器。六边形架构的实际收益是：你可以在不启动 Spring、不连接数据库的情况下，对领域模型进行全面的单元测试。领域逻辑的正确性不依赖于任何基础设施。</p>
<h3>贫血领域模型：最常见的战术设计反模式</h3>
<p>所谓贫血领域模型，是指实体只有数据属性和 getter/setter，所有业务逻辑都写在 Service 层：</p>
<pre><code class="language-java">// 贫血模型：Order 退化为数据容器
public class Order {
    private Long id;
    private String status;
    private BigDecimal totalAmount;
    private List&lt;OrderLine&gt; lines;
    // 只有 getter 和 setter，没有业务行为
}

// 所有逻辑堆在 Service 中
public class OrderService {
    public void addLine(Long orderId, ProductId productId, int qty, Money price) {
        Order order = orderRepo.findById(orderId);
        if (!&quot;DRAFT&quot;.equals(order.getStatus())) {
            throw new RuntimeException(&quot;...&quot;);
        }
        OrderLine line = new OrderLine(productId, qty, price);
        order.getLines().add(line);  // 直接操作内部集合
        BigDecimal total = order.getLines().stream()...
        order.setTotalAmount(total);
        orderRepo.save(order);
    }
}
</code></pre>
<p>这种写法的问题在于：不变量（总金额 = Σ 行金额）没有被聚合保护，任何地方都可以直接 setTotalAmount 而不更新行项，或者直接操作 lines 集合而不重算总额。当 OrderService 之外的另一个 Service 也需要修改 Order 时，不变量的维护就变成了&quot;记得调用&quot;而非&quot;编译器保证&quot;。</p>
<p>对比之下，前面&quot;聚合设计实战&quot;中的 Order 实现，addLine 方法内部自动 recalculateTotal，外部根本没有 setTotalAmount 方法可调——不变量由聚合自身保护。</p>
<h2>微服务间的通信与一致性</h2>
<h3>从单体到微服务的一致性挑战</h3>
<p>在单体应用中，多个聚合在同一个进程边界内，可以在一个事务中完成：客户下单 → 扣减库存 → 发送邮件。所有操作要么都成功，要么都失败。</p>
<p>但微服务化后，这些聚合分散到了不同的分布式系统中。根据 <strong>CAP 定理</strong>，一个分布式系统只能同时满足一致性（C）、可用性（A）、分区容错（P）中的两个。在现实系统中，分区容错是不可协商的——网络不可靠、虚拟机可以宕机、区域延迟可能恶化。因此我们只能在可用性和一致性之间选择。而在现代互联网应用中，牺牲可用性通常也不可接受。<strong>结论：基于最终一致性设计应用程序。</strong></p>
<h3>领域事件设计</h3>
<p>领域事件（Domain Event）是实现最终一致性的核心载体。一个设计良好的领域事件应该包含足够的信息让消费者独立处理，而不需要回调生产者查询更多数据：</p>
<pre><code class="language-java">public record OrderCancelled(
    String eventId,                           // 全局唯一事件 ID，用于消费端幂等去重
    OrderId orderId,
    CustomerId customerId,
    Money orderAmount,
    String reason,
    List&lt;CancelledLineItem&gt; cancelledItems,  // 包含足够的信息让库存服务恢复库存
    Instant occurredAt,
    long eventVersion                         // 用于乱序处理
) implements DomainEvent {

    public record CancelledLineItem(ProductId productId, int quantity) {}
}
</code></pre>
<p>事件驱动架构避免了两种耦合：行为耦合（一个领域无需规定其他领域应该做什么）和时间耦合（一个流程的完成不依赖于所有系统同时可用）。</p>
<h3>Outbox 模式：保证事件发布的可靠性</h3>
<p>事件驱动架构面临一个经典难题：聚合状态的持久化和事件的发布是两个操作——如果数据库写入成功但消息队列发送失败，就会出现数据和事件不一致。</p>
<p>Outbox 模式的解决思路是利用数据库事务的原子性：在同一个事务中，既写入聚合状态，又将事件写入同一数据库的 Outbox 表。然后由一个独立的进程（Relay/Poller 或基于 CDC 的 Debezium）从 Outbox 表读取事件并发布到消息队列。</p>
<pre><code class="language-sql">-- 在同一个事务中
BEGIN;
  UPDATE orders SET status = &#39;CANCELLED&#39;, updated_at = NOW() WHERE id = ?;
  INSERT INTO outbox (aggregate_type, aggregate_id, event_type, payload, created_at)
    VALUES (&#39;Order&#39;, ?, &#39;OrderCancelled&#39;, ?::jsonb, NOW());
COMMIT;
</code></pre>
<p>这样即使消息队列暂时不可用，事件也不会丢失——它安全地保存在数据库中，等待重试发布。</p>
<h3>事件消费端的可靠性：幂等与乱序</h3>
<p>Outbox 解决了生产端的可靠性，但消费端同样面临挑战。Outbox Relay 可能重复发送同一条事件（at-least-once 语义），网络或队列分区可能导致事件乱序到达。消费端必须做好两件事：</p>
<p><strong>幂等消费</strong>：同一事件重复到达不产生副作用。常见做法是用事件 ID 做去重表：</p>
<pre><code class="language-java">@Transactional
public void handle(OrderCancelled event) {
    // 幂等检查：如果这条事件已经处理过，直接返回
    if (processedEventRepo.existsById(event.eventId())) {
        return;
    }

    // 执行业务逻辑
    for (var item : event.cancelledItems()) {
        inventory.restoreStock(item.productId(), item.quantity());
    }

    // 记录已处理，与业务操作在同一事务中
    processedEventRepo.save(new ProcessedEvent(event.eventId(), Instant.now()));
}
</code></pre>
<p><strong>乱序处理</strong>：事件可能不按发生顺序到达。消费者可以利用事件中的 eventVersion 或时间戳判断：如果收到的事件版本比本地记录的版本更旧，说明是迟到的旧事件，可以安全忽略。</p>
<table>
<thead>
<tr>
<th>角色</th>
<th>保障措施</th>
</tr>
</thead>
<tbody><tr>
<td><strong>生产者</strong></td>
<td>Outbox 模式确保事件<strong>至少发出一次</strong></td>
</tr>
<tr>
<td><strong>消费者</strong></td>
<td>基于事件 ID 去重，以<strong>幂等方式</strong>消费</td>
</tr>

<td><strong>乱序处理</strong></td>
<td>基于 eventVersion 或时间戳，忽略迟到的旧事件</td>
</tr>
</tbody></table>
<h3>Saga 模式：跨服务的业务流程编排</h3>
<p>当一个业务流程需要跨多个微服务协调（如下单涉及订单、库存、支付三个服务），单个事务无法覆盖，就需要 Saga。Saga 将长事务拆分为一系列本地事务，每个本地事务对应一个补偿操作（Compensating Action），如果某个步骤失败，按反序执行补偿操作来回滚之前的步骤。</p>
<p>Saga 有两种实现方式：</p>
<table>
<thead>
<tr>
<th>方式</th>
<th>机制</th>
<th>适用场景</th>
<th>风险</th>
</tr>
</thead>
<tbody><tr>
<td><strong>编排式（Choreography）</strong></td>
<td>每个服务监听上一步的事件，决定自己是执行还是补偿</td>
<td>步骤少（2-3 步）、参与者少</td>
<td>步骤多时流程难以追踪</td>
</tr>
<tr>
<td><strong>协调式（Orchestration）</strong></td>
<td>由一个 Saga 协调器集中管理流程状态和步骤推进</td>
<td>步骤多、需要可视化流程</td>
<td>协调器本身成为关键节点</td>
</tr>
</tbody></table>
<p>以协调式 Saga 为例，下单流程的每个步骤都定义正向操作和补偿操作：</p>
<pre><code class="language-java">public class CreateOrderSaga {
    private final List&lt;SagaStep&gt; steps = List.of(
        new SagaStep(&quot;reserveInventory&quot;, this::reserveInventory, this::releaseInventory),
        new SagaStep(&quot;processPayment&quot;,   this::processPayment,   this::refundPayment),
        new SagaStep(&quot;confirmOrder&quot;,     this::confirmOrder,     this::rejectOrder)
    );

    public void execute(CreateOrderCommand cmd) {
        SagaState state = SagaState.start(cmd);

        for (SagaStep step : steps) {
            try {
                step.execute(state);
                state.markCompleted(step.name());
            } catch (Exception e) {
                // 失败：按反序执行已完成步骤的补偿操作
                state.completedSteps().reversed().forEach(
                    completedName -&gt; findStep(completedName).compensate(state)
                );
                state.markFailed(step.name(), e);
                return;
            }
        }
        state.markSucceeded();
    }
}
</code></pre>
<p>关键在于：每个步骤的正向操作和补偿操作都是本地事务，通过 Outbox 模式保证可靠发布。补偿操作不是&quot;撤销&quot;，而是一个语义上的逆操作——比如&quot;释放库存&quot;而不是&quot;删除库存预留记录&quot;。</p>
<h3>CQRS：读写模型分离</h3>
<p>Command Query Responsibility Segregation（CQRS）将系统的读模型和写模型分离：写操作通过聚合根维护业务不变量，读操作通过专门的查询模型提供高性能查询。</p>
<p>CQRS 不是所有微服务都需要的——它适用于<strong>读写差异大</strong>的场景。例如：订单服务的写模型是 Order 聚合（强一致性、完整的业务规则），但运营后台的订单列表页需要关联商品名称、客户信息、物流状态等多个聚合的数据。如果每次查询都通过聚合根加载再拼装，性能和复杂度都不可接受。</p>
<p>CQRS 的做法是维护一个为列表查询优化的只读视图，通过领域事件异步更新：</p>
<pre><code class="language-java">// 读模型：为查询优化的扁平化视图
public class OrderSummaryView {
    private String orderId;
    private String customerName;    // 冗余存储，避免查询时 join Customer
    private String statusDisplay;
    private BigDecimal totalAmount;
    private LocalDateTime createdAt;
    private String latestTrackingNo; // 冗余存储，避免查询时 join Fulfillment
}

// 事件处理器：监听领域事件，更新读模型
@Component
public class OrderSummaryProjection {

    @EventHandler
    public void on(OrderSubmitted event) {
        Customer customer = customerQueryService.findById(event.customerId());
        OrderSummaryView view = new OrderSummaryView();
        view.setOrderId(event.orderId().value());
        view.setCustomerName(customer.getName());
        view.setStatusDisplay(&quot;已提交&quot;);
        view.setTotalAmount(event.totalAmount().amount());
        view.setCreatedAt(event.occurredAt());
        summaryViewRepo.save(view);
    }

    @EventHandler
    public void on(OrderShipped event) {
        summaryViewRepo.updateTracking(event.orderId().value(), event.trackingNo());
        summaryViewRepo.updateStatus(event.orderId().value(), &quot;已发货&quot;);
    }
}
</code></pre>
<p>读模型和写模型之间存在短暂的数据延迟（最终一致性），这在大多数查询场景下是可以接受的。但如果业务要求提交后立刻在列表中看到最新状态，可以在写操作返回后同步更新读模型，或者在前端做乐观更新。</p>
<h3>何时仍需同步调用？</h3>
<p>并非所有场景都适合事件驱动。当需要<strong>即时反馈</strong>时（如购物车→支付授权），仍需同步 API 调用。但要注意：同步调用引入了行为耦合和时间耦合，被调用服务不可用时调用方也会受影响。</p>
<p><strong>缓解策略</strong>：同步调用作为主路径，辅以基于事件或批处理的异步重试作为降级方案。在用户体验、系统弹性和运营成本之间做好权衡。</p>
<h2>BFF 模式：解耦前端与领域服务</h2>
<h3>问题：服务为了迎合调用者而变形</h3>
<p>微服务架构中一个常见的反模式是：<strong>域服务为了满足前端的特定数据需求而编排其他服务</strong>。</p>
<p>以&quot;订单详情页&quot;为例，页面需要同时展示订单信息和退款信息。如果让订单服务调用退款服务来组装复合响应：订单服务的自治性降低、增加故障点、变更成本高。</p>
<h3>解决方案：Backend for Frontends（BFF）</h3>
<p>BFF 是由<strong>消费者团队</strong>（前端团队）创建和维护的后端服务，负责对多个域服务进行集成和编排、为前端提供定制化的数据契约、根据不同终端优化响应格式和体积。</p>
<table>
<thead>
<tr>
<th>对比</th>
<th>无 BFF</th>
<th>有 BFF</th>
</tr>
</thead>
<tbody><tr>
<td>数据编排</td>
<td>域服务互相调用，或前端直接调多个服务</td>
<td>BFF 统一编排，域服务保持纯粹</td>
</tr>
<tr>

<td>前端需求变化要改多个域服务</td>

<h2>常见反模式与陷阱</h2>
<p>DDD 微服务实践中有几个高频踩坑点，值得单独拎出来。</p>
<h3>分布式单体</h3>
<p>表面上拆成了多个服务，但所有服务共享同一个数据库，或者每次变更必须同时部署多个服务——这不是微服务，而是分布式单体。它继承了单体的所有缺点（无法独立部署、无法独立扩展），同时还增加了分布式系统的复杂性（网络延迟、分布式事务、运维成本）。</p>
<p><strong>症状</strong>：部署一个服务需要同时部署另外三个；两个服务的数据库表之间有外键约束；改一个服务的 API 需要另一个服务同步跟改。</p>
<h3>按技术层拆分而非按业务域拆分</h3>
<p>把用户相关的表放在一个&quot;用户服务&quot;、把所有 API 网关逻辑放在&quot;网关服务&quot;、把所有消息处理放在&quot;消息服务&quot;——这是按技术关注点拆分，不是按业务域拆分。结果是每个业务变更都需要改多个服务。</p>
<p>正确的做法是：每个服务对应一个界限上下文或聚合，包含从 API 到数据库的完整垂直切面。</p>
<h3>数据所有权不清</h3>
<p>两个服务同时写同一张表，或者一个服务直接读另一个服务的数据库——这是数据所有权不清的典型表现。解决方案是：每个聚合的数据只由其所属服务管理，其他服务需要数据时通过 API 或事件获取。</p>
<h3>过早拆分</h3>
<p>在业务模型还没有稳定的时候就急于拆分为微服务，结果频繁的需求变更导致服务边界不断调整，重构成本远超单体。<strong>如果你还不确定边界在哪里，就不要拆分。</strong> 先在单体内通过模块边界和接口隔离做好&quot;可拆分&quot;的准备，等业务稳定后再拆分。</p>
<h2>渐进式拆分路线图</h2>
<p>将以上所有工具整合，从单体拆分到微服务的推荐路径：</p>
<h3>第一步：战略设计</h3>
<p>与领域专家一起梳理业务，划分子域并识别核心域、支撑域和通用域。为每个子域确定界限上下文的边界，在每个上下文内建立一致的统一语言。</p>
<h3>第二步：协作建模</h3>
<p>通过事件风暴跨团队协作，识别领域事件、命令、聚合和热点问题。在此基础上绘制上下文映射，明确上下文之间的依赖关系和协作模式。</p>
<h3>第三步：服务划分</h3>
<p>根据聚合和上下文映射，确定每个微服务的边界。区分同步调用和异步事件驱动的通信方式，为不同终端规划 BFF 层。</p>
<h3>第四步：渐进式拆分</h3>
<p>从边缘开始，先拆分耦合最少、边界最清晰的服务。采用<strong>绞杀者模式（Strangler Fig Pattern）</strong>：新功能用微服务实现，老功能逐步迁移。每拆分一个服务，验证边界是否正确，必要时调整。</p>
<h3>第五步：战术深化</h3>
<p>对核心域服务实施完整的战术设计——聚合、Entity、Value Object、Domain Service、Repository、Application Service 分层组织。对支撑域和通用域，根据复杂度决定是否需要完整的战术设计，还是简单的 CRUD 即可。</p>
<h2>总结</h2>
<p>基于 DDD 构建微服务，战略设计和战术设计缺一不可。战略设计决定<strong>边界在哪里</strong>——子域分类告诉你资源优先级，事件风暴发现聚合和事件流，上下文映射定义服务间的协作契约。战术设计决定<strong>边界内怎么建</strong>——聚合保护业务不变量，Entity 和 Value Object 构建领域模型，Repository 和 Application Service 将领域逻辑与基础设施解耦。跨服务的一致性则由 Outbox 保证事件可靠发布、Saga 编排跨服务流程、CQRS 分离读写模型。</p>
<p>几条核心原则值得反复校验：微服务的本质是界限清晰，不是规模小；聚合是数据一致性的原子边界，设计小聚合，通过事件实现跨聚合的最终一致性；上下文映射不只是画图，而是在做架构决策；先在单体内做好模块隔离，再渐进式拆分——合并两个服务的成本远高于拆分一个。</p>
<blockquote>
<p>DDD 不是银弹，它是一种思考方式。它引导我们从业务本质出发，用结构化的方法找到正确的服务边界。在微服务架构中，<strong>找到正确的边界比选择正确的技术栈重要十倍</strong>。</p>
</blockquote>
1f:T836b,<p>单 Agent 系统迟早会撞上四个天花板——上下文装不下、能力难以兼顾、串行执行慢、决策视角单一。撞上时人们的反应通常是&quot;拆成多个 Agent&quot;——但这一步比想象中危险得多。串联两个成功率 80% 的 Agent，端到端只剩 64%；三个串联就是 51%。<strong>多 Agent 不是免费升级，是用乘法效应换专业化</strong>。这一篇展开四种协作模式的伪代码骨架、通信机制的 schema 选型、失败传播的分层处理逻辑，以及一个比&quot;思路对不对&quot;更重要的问题——单节点可靠性够不够。</p>

<p>但<strong>先把丑话说在前面</strong>：单 Agent 任务成功率 80% 意味着两个串联 Agent 系统 64%，三个串联 51%。<strong>多 Agent 架构必须在每个节点引入校验和容错，否则数学上跑不通</strong>——这是后续讨论的所有协作模式的共同约束。</p>
<p>这个数学事实有一个推论：<strong>多 Agent 系统的可靠性目标不能只看终端</strong>，必须分解到每个节点。如果整个系统要达到 95% 端到端成功率，三个节点串联意味着每个节点要 95%^(1/3) ≈ 98.3%。这比单节点 95% 要求高得多，工程难度也大得多。<strong>很多多 Agent 项目失败的根因不是思路错，而是对单节点可靠性要求理解不足</strong>。</p>
<pre><code class="language-python">def required_per_node_reliability(target: float, n_nodes: int) -&gt; float:
    &quot;&quot;&quot;端到端目标 target、N 个节点串联，每个节点至少要多少可靠性&quot;&quot;&quot;
    return target ** (1 / n_nodes)

# 端到端 95% / 3 个节点 → 98.3%
# 端到端 99% / 3 个节点 → 99.66%
# 端到端 99% / 5 个节点 → 99.80%
# 节点越多、目标越高，单节点要求越极端
</code></pre>
<p>这个公式有一个反直觉的含义：<strong>节点越多，单节点要求越高，但单节点能做到的可靠性是有天花板的</strong>。LLM 单步 99% 已经是工程极限——意味着 5 节点串联端到端只能做到约 95%。要更高的端到端可靠性，必须加非串联结构（投票、冗余、人审）打破乘法效应。</p>

<h2>2. 四种协作模式</h2>

<h3>2.1 Supervisor-Worker：中央协调</h3>
<p><img src="/images/blog/agentic/supervisor-worker.svg" alt="Supervisor-Worker"></p>
<p>中央协调者（Supervisor）分解任务、分配给专门的 Worker，最后合成结果。Worker 之间不直接通信。</p>
<pre><code class="language-python">def supervisor_worker(goal: str, workers: dict[str, Agent]) -&gt; Result:
    &quot;&quot;&quot;Supervisor 分解任务、并行调 Worker、合成结果&quot;&quot;&quot;
    # 1. Supervisor 分解
    plan = supervisor.complete(
        DECOMPOSE_PROMPT.format(goal=goal, workers=workers.keys()),
        schema=WORK_ASSIGNMENT_SCHEMA,
    )
    # plan.assignments = [{&quot;worker&quot;: &quot;search&quot;, &quot;task&quot;: &quot;...&quot;}, ...]

    # 2. 并行调 Worker（互不依赖时）
    results = {}
    with asyncio.TaskGroup() as tg:
        for assignment in plan.assignments:
            worker = workers[assignment.worker]
            tg.create_task(
                run_worker_with_retry(worker, assignment, results)
            )

    # 3. 合成
    return supervisor.complete(
        SYNTHESIZE_PROMPT.format(plan=plan, results=results),
    )

async def run_worker_with_retry(worker, assignment, results):
    &quot;&quot;&quot;每个 Worker 调用都自带重试/降级链&quot;&quot;&quot;
    for attempt in range(3):
        try:
            r = await worker.run(assignment.task, timeout=assignment.timeout)
            if validate(r, schema=assignment.expected_schema):
                results[assignment.id] = r
                return
        except WorkerError as e:
            if not is_retryable(e):
                break
    # 主 Worker 三次失败 → 走降级
    fallback = workers.get(assignment.fallback)
    if fallback:
        results[assignment.id] = await fallback.run(assignment.task)
    else:
        results[assignment.id] = Result.failed(reason=&quot;no fallback&quot;)

<h3>2.2 Peer-to-Peer：平等协商</h3>
<p><img src="/images/blog/agentic/peer-to-peer.svg" alt="Peer-to-Peer"></p>
<p>多个 Agent 地位平等，直接消息传递，无中央协调。最常见用法是&quot;辩论&quot;模式：</p>
<pre><code class="language-python">def peer_to_peer_debate(
    topic: str,
    agents: list[Agent],
    max_rounds: int = 5,
    converge_threshold: float = 0.85,
) -&gt; Result:
    &quot;&quot;&quot;N 个 Agent 轮流发言，直到达成共识或超过轮次&quot;&quot;&quot;
    transcript = [Message(speaker=&quot;moderator&quot;, content=topic)]

    for round_idx in range(max_rounds):
        round_messages = []
        for agent in agents:
            response = agent.complete(
                DEBATE_PROMPT.format(
                    role=agent.role,
                    transcript=transcript[-20:],   # 滑动窗口防止 context 爆
                    round=round_idx,
                ),
                schema=DEBATE_TURN_SCHEMA,
            )
            round_messages.append(Message(speaker=agent.id, **response))

        transcript.extend(round_messages)

        # 关键：必须有外部终止机制
        if detect_consensus(round_messages, threshold=converge_threshold):
            return synthesize_consensus(transcript)
        if detect_loop(transcript, window=2 * len(agents)):
            return arbitrate(transcript)  # 引入外部仲裁打破僵局

    # 超过 max_rounds 还没达成共识 → 强制仲裁
    return arbitrate(transcript)

def detect_consensus(round_messages, threshold=0.85) -&gt; bool:
    &quot;&quot;&quot;共识检测的三种典型实现，按强度排序&quot;&quot;&quot;
    # 方案 1：所有 Agent 的 confidence × position 余弦相似度 &gt; 阈值
    positions = [m.position for m in round_messages]
    sim_matrix = [[cosine(embed(a), embed(b)) for b in positions] for a in positions]
    pairwise_avg = sum(s for row in sim_matrix for s in row) / (len(positions) ** 2)
    if pairwise_avg &gt; threshold:
        return True
    # 方案 2：用一个独立的 LLM judge 判断&quot;这些立场是否实质上一致&quot;
    judge_result = judge_llm.complete(JUDGE_CONSENSUS_PROMPT.format(positions=positions))
    return judge_result.consensus and judge_result.confidence &gt; 0.8

def arbitrate(transcript) -&gt; Result:
    &quot;&quot;&quot;三种打破僵局的方式&quot;&quot;&quot;
    # 1) 投票多数派——Agent 间存在不可调和分歧时最朴素
    # 2) 引入更高权重的&quot;仲裁 Agent&quot;做最终裁决
    # 3) 直接升级到人审——所有自动化都做不了决定时的兜底
    return weighted_vote(transcript) or escalate_to_human(transcript)

<h3>2.3 Pipeline：流水线</h3>
<p><img src="/images/blog/agentic/pipeline.svg" alt="Pipeline"></p>
<p>Agent 按顺序串联，上游输出是下游输入，数据单向流动。</p>
<pre><code class="language-python">def pipeline_with_checkpoint(input_data, stages: list[Stage]) -&gt; Result:
    &quot;&quot;&quot;带 checkpoint 的 Pipeline：每阶段输出落盘，失败可从最近 checkpoint 恢复&quot;&quot;&quot;
    state = PipelineState(input=input_data, outputs={}, current_stage=0)
    state = load_checkpoint_if_exists(state.run_id) or state

    for i in range(state.current_stage, len(stages)):
        stage = stages[i]
        try:
            # 准备输入：当前阶段拿前一阶段的输出
            stage_input = state.outputs.get(stages[i - 1].name) if i &gt; 0 else input_data
            result = stage.run(stage_input)

            if not validate(result, schema=stage.output_schema):
                # 校验失败 → 视为本阶段失败
                raise StageValidationError(stage.name, result)

            state.outputs[stage.name] = result
            state.current_stage = i + 1
            save_checkpoint(state)            # 每阶段成功后立刻 checkpoint

        except StageError as e:
            if stage.fallback:
                # 走降级：跳过本阶段，把前阶段输出直接传给下下阶段
                state.outputs[stage.name] = stage.fallback(stage_input)
                save_checkpoint(state)
                continue
            return Result.failed(stage=stage.name, error=e, partial_state=state)

    return Result.ok(outputs=state.outputs)

<td>Dynamic Routing</td>
</tr>
</tbody></table>
<p>大多数任务最自然的形态是 Supervisor-Worker——&quot;一个总目标，分解后并行执行，最后合成&quot;。从它开始不容易选错。</p>
<p><strong>四种模式不是互斥的</strong>。生产中的复杂多 Agent 系统通常是混合形态：</p>
<ul>
<li>外层是 Dynamic Routing 把请求按类型分发</li>
<li>每个路由分支内部是 Supervisor-Worker 做任务分解</li>
<li>关键决策节点内嵌 Peer-to-Peer 做辩论式分析</li>
<li>输出生成阶段走 Pipeline 做&quot;草稿→审查→润色&quot;</li>
</ul>
<p>把多 Agent 看成&quot;模式的组合&quot;而非&quot;单一模式&quot;，能更好地匹配真实任务的复杂性。但越多模式叠加，调试难度增长越快——所以混合形态应该是被现实推出来的，不是被设计欲推出来的。</p>
<h3>2.6 四种模式都不太合适怎么办</h3>
<p>如果四种都尝试过仍然撞墙，往往意味着两件事之一：<strong>任务还不够独立 = 多 Agent 救不了，回到单 Agent + 工作流编排</strong>，或<strong>单节点可靠性不够 = 拆 Agent 只会让乘法效应更糟</strong>。这时三个降级路径按收益排序：</p>
<ol>
<li><strong>回到单 Agent，但加工作流编排</strong>——很多看起来需要多 Agent 的场景，其实只需要更好的 Routing + Parallelization</li>
<li><strong>把多 Agent 退化为工具化的 sub-Agent</strong>——子 Agent 作为主 Agent 的特殊工具，主从关系明确，比对等协作好调试</li>
<li><strong>承认这个任务现阶段不该自动化</strong>——HITL 兜底 + 沉淀人类决策模式，等真实数据足够再考虑自动化</li>
</ol>
<p>第三条最被低估——多 Agent 项目最大的失败模式是&quot;不该自动化的事强行自动化&quot;。</p>

<h2>3. 黑板、消息、事件：Agent 之间怎么传信息</h2>

<pre><code class="language-python">Blackboard = {

    &quot;goal&quot;: str,
    &quot;shared_facts&quot;: dict,           # 各 Agent 共享的事实库

<h2>4. 状态分区：避免冲突的核心思路</h2>
<p>Multi-Agent 系统的状态管理比 Single-Agent 复杂一个数量级。多个 Agent 同时操作状态，怎么保证一致性？</p>
<table>
<thead>
<tr>
<th>共享状态</th>
<th>独立状态</th>
</tr>
</thead>
<tbody><tr>
<td>Agent 间信息同步即时</td>
<td>无并发问题</td>
</tr>
<tr>
<td>任何 Agent 看到最新全局状态</td>
<td>每个 Agent 完全自主</td>
</tr>
<tr>
<td>需要处理并发冲突</td>
<td>Agent 间信息同步有延迟，需要显式合并</td>
</tr>
<tr>
<td>适合 Supervisor-Worker</td>
<td>适合 Pipeline</td>
</tr>
</tbody></table>
<p><strong>最有效的冲突避免不是冲突解决</strong>——是<strong>状态分区</strong>。每个 Agent 只写自己负责的状态区域，避免多 Agent 写同一个 key。Supervisor-Worker 天然支持这个：每个 Worker 写自己的结果 key，只有 Supervisor 读所有 key。</p>
<pre><code class="language-python">def write_partition(blackboard, agent_id, key, value):
    &quot;&quot;&quot;每个 Agent 只能写自己的分区——一行权限校验防住绝大多数冲突&quot;&quot;&quot;
    assert key.startswith(f&quot;agent_outputs.{agent_id}.&quot;), \
        f&quot;agent {agent_id} cannot write to {key}&quot;
    blackboard[key] = value
</code></pre>
<p>这种&quot;权限分区&quot;在工程上比&quot;加锁&quot;简单得多——锁机制处理不好就死锁、性能损失大；分区是设计层面避免冲突，几乎零成本。</p>
<hr>
<h2>5. 失败在哪个层面发生：Worker、Supervisor、Pipeline</h2>
<p>Multi-Agent 的错误传播路径比单 Agent 更多。三层处理：</p>
<h3>5.1 Worker 失败</h3>
<p>按优先级处理：</p>
<ol>
<li><strong>同一 Worker 重试</strong>——瞬时错误（网络、API 限流）</li>
<li><strong>降级到备选 Worker</strong>——主 Worker 持续失败时切换。需要预先配置降级链</li>
<li><strong>跳过 + 上报</strong>——非关键 Worker 失败时跳过，让流程继续</li>
<li><strong>整体失败 + 告知用户</strong>——关键 Worker 不可恢复时止损</li>
</ol>
<p>降级链的配置 schema：</p>
<pre><code class="language-python">WorkerConfig = {
    &quot;id&quot;: &quot;search_agent&quot;,
    &quot;primary&quot;: {&quot;model&quot;: &quot;gpt-4o&quot;, &quot;tools&quot;: [&quot;google_search&quot;, &quot;web_scrape&quot;]},
    &quot;fallback_chain&quot;: [
        {&quot;model&quot;: &quot;gpt-4o-mini&quot;, &quot;tools&quot;: [&quot;google_search&quot;]},   # 弱化版
        {&quot;model&quot;: &quot;haiku-4-5&quot;, &quot;tools&quot;: [&quot;cached_search&quot;]},     # 仅缓存
    ],
    &quot;max_retries_per_level&quot;: 2,
    &quot;is_critical&quot;: False,                  # 关键性标志：False 时失败可跳过
}
</code></pre>
<h3>5.2 Supervisor 失败</h3>
<p>Supervisor 是中央协调者，失败意味着整个任务无法继续。三层保护：</p>
<ul>
<li><strong>外部监控</strong>：非 LLM 的监控层检测 Supervisor 健康状态</li>
<li><strong>Supervisor 冗余</strong>：备用 Supervisor（可用不同模型），主切备</li>
<li><strong>Checkpoint 机制</strong>：每个决策点保存状态快照，从最近的 checkpoint 恢复</li>
</ul>
<h3>5.3 Pipeline 链路中断</h3>
<table>
<thead>
<tr>
<th>策略</th>
<th>做法</th>
<th>适用</th>
</tr>
</thead>
<tbody><tr>
<td>重试当前节点</td>
<td>相同输入重新执行</td>
<td>瞬时故障</td>
</tr>
<tr>
<td>跳过 + 降级</td>
<td>跳过失败节点，前节点输出直接给下下节点</td>
<td>失败节点是可选增强步骤（翻译、润色）</td>
</tr>
<tr>
<td>Checkpoint 回退</td>
<td>缓存每个节点的输入输出，从失败处重新开始</td>
<td>链路长、前面步骤成本高</td>
</tr>
</tbody></table>
<h3>5.4 Peer-to-Peer 的死锁</h3>
<p>两个 Agent 互相等待对方的回复——A 等 B 的反馈，B 等 A 的修改。检测策略：维护&quot;等待图&quot;，定期检测是否有环。出现环时强制超时一方。</p>
<pre><code class="language-python">def detect_deadlock(wait_graph: dict[str, set[str]]) -&gt; list[str] | None:
    &quot;&quot;&quot;wait_graph[A] = {B, C} 表示 A 在等 B 和 C&quot;&quot;&quot;
    visited, stack = set(), []
    def dfs(node):
        if node in stack:
            return stack[stack.index(node):]  # 找到环
        if node in visited:
            return None
        visited.add(node)
        stack.append(node)
        for nb in wait_graph.get(node, []):
            if cycle := dfs(nb):
                return cycle
        stack.pop()
        return None

    for n in wait_graph:
        if cycle := dfs(n):
            return cycle
    return None
</code></pre>
<p>发现环时的处理：强制超时一方（按优先级最低或资源消耗最大的 Agent）、把僵局上报给上级 Supervisor 或人审。</p>
<p>为什么不直接用 NetworkX 这类成熟拓扑排序库？两个原因：<strong>实时检测的开销</strong>——Agent 间的 wait_graph 在每轮交互后都可能变化，NetworkX 的图构造开销在高频场景下不划算；<strong>等待语义的特殊性</strong>——传统拓扑排序处理的是&quot;任务 A 必须在 B 之前&quot;的硬依赖，Agent 死锁是&quot;A 在等 B 的反馈、B 也在等 A 的修改&quot;的双向软依赖，需要专门的环检测逻辑。一旦 wait_graph 复杂到 100+ 节点，再考虑引入第三方库。</p>

<h2>7. Multi-Agent 调试的平方级难度</h2>
<p>Multi-Agent 调试难度是 Single-Agent 的平方级增长——不仅每个 Agent 内部可能出错，Agent 之间的交互也可能出错。</p>
<p>可观测性的基本要求：</p>
<table>
<thead>
<tr>
<th>维度</th>
<th>记录什么</th>
</tr>
</thead>
<tbody><tr>
<td>Trace</td>
<td>每个 Agent 的每次 LLM 调用、输入、输出、耗时</td>
</tr>
<tr>
<td>Span 嵌套</td>
<td>Agent 之间的调用关系——父 Span（Supervisor）下嵌套子 Span（Worker）</td>
</tr>
<tr>
<td>状态快照</td>
<td>每个决策点的 Blackboard 状态</td>
</tr>
<tr>
<td>消息日志</td>
<td>Agent 间所有消息的完整内容</td>
</tr>
</tbody></table>
<h3>Trace span 的最小 schema</h3>
<pre><code class="language-python">AgentSpan = {
    &quot;span_id&quot;: str,
    &quot;parent_span_id&quot;: str | None,   # Supervisor span 是父，Worker span 是子
    &quot;agent_id&quot;: str,
    &quot;operation&quot;: &quot;plan&quot; | &quot;execute&quot; | &quot;reflect&quot; | &quot;synthesize&quot;,
    &quot;input&quot;: dict,
    &quot;output&quot;: dict | None,
    &quot;model&quot;: str,
    &quot;tokens&quot;: {&quot;prompt&quot;: int, &quot;completion&quot;: int},
    &quot;duration_ms&quot;: int,
    &quot;status&quot;: &quot;ok&quot; | &quot;retry&quot; | &quot;failed&quot; | &quot;fallback_used&quot;,
    &quot;error&quot;: str | None,
    &quot;blackboard_snapshot_ref&quot;: str | None,   # 指向当时 BB 的快照
}
</code></pre>
<p><strong>特别难的是 bug 复现</strong>：</p>
<ul>
<li>LLM 输出非确定（相同输入可能产生不同输出）</li>
<li>Agent 间交互动态（执行路径取决于中间结果）</li>
<li>并发时序不确定（Worker A 和 B 谁先完成可能影响结果）</li>
</ul>
<p>应对：记录每次 LLM 调用的完整 messages 和 response（不只是摘要）；用固定 seed 和 temperature=0 做确定性回放；快照式调试——在每个 Agent 决策点保存完整 Blackboard 快照。</p>

<p>这六条不是 checklist，是&quot;任一不过都该停手&quot;的硬门槛。多 Agent 项目最大的风险不是技术失败，是<strong>上线后才发现根本没有调试和兜底能力</strong>——线上出错时既看不清状态、又改不动 prompt、又没法回滚。</p>
<hr>
<h2>9. 从单到多不是相加，是相乘</h2>
<p>Multi-Agent 系统最反直觉的一件事是数学：单 Agent 任务成功率 80% 意味着两个串联系统是 64%、三个是 51%。要让三个串联系统达到 95% 端到端可靠性，每个单节点必须达到约 98.3%——这是一个比单 Agent 95% 严格得多的工程要求。<strong>很多 Multi-Agent 项目失败的根因不是思路错，而是低估了单节点可靠性的要求</strong>。</p>
<p>Multi-Agent 解决的是单 Agent 的四个天花板：上下文不够、能力单一、串行执行、决策视角单一。换来的代价也是四件：2-3 倍 token 成本、平方级调试难度、新的协作错误模式（Agent 间死锁、信息不同步、状态竞争）、新的运维问题（Supervisor 单点失败、Worker 降级链路）。是用复杂度换能力上限，不是免费午餐。</p>
<p>最实用的判断是这样：先穷尽单 Agent 的优化空间——更好的 prompt、更精细的工具、更强的模型、更聪明的 Memory——再考虑 Multi-Agent。如果你的单 Agent 还在 70% 成功率上挣扎，拆成三个 Agent 只会让端到端可靠性更差。Multi-Agent 是&quot;已经做到 90% 想冲 95%&quot;的工具，不是&quot;做不到 70% 想跳到 80%&quot;的工具。</p>
