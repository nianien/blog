<p>本文是信息流系列的下篇。上篇<a href="/blog/engineering/architecture/%E7%99%BE%E4%B8%87%E7%BA%A7%E5%86%85%E5%AE%B9%E7%A4%BE%E5%8C%BA%E4%BF%A1%E6%81%AF%E6%B5%81%EF%BC%88%E4%B8%8A%EF%BC%89%EF%BC%9A%E6%8A%80%E6%9C%AF%E6%9E%B6%E6%9E%84%E5%85%A8%E8%A7%A3">《百万级内容社区信息流（上）：技术架构全解》</a>已经完整拆解了从内容生产、特征工程、召回排序到 Feed 分发的全链路技术架构。本文聚焦架构之上的<strong>运营体系、商业化路径、隐私合规、技术创新方向和团队建设</strong>，目标读者是技术产品经理、增长/运营负责人，以及关心&quot;技术如何服务业务&quot;的工程师。</p>

<p>抖音的&quot;流量池赛马&quot;是目前最被广泛模仿的方案。上篇第 5.3 节从技术角度拆解了流量池的工程实现（Boost 因子控制、与 AB 实验联动），这里补充运营视角的关键设计决策：</p>
<p><strong>晋升阈值的设定</strong>：不同类目的阈值不同——知识类内容完播率天然低于娱乐类，不能用统一阈值。实际操作中按类目分组设定，且阈值本身作为 AB 实验参数持续调优。<strong>观察窗口</strong>也因内容类型而异：短视频在初始池的观察窗口是 2-4 小时（生命周期短），图文笔记可以拉长到 12-24 小时（长尾消费更明显）。<strong>淘汰并非终局</strong>——未通过初始池的内容不会被删除，而是沉入长尾召回池，仍有可能通过搜索或关注流被消费。部分平台还提供&quot;复活&quot;机制：创作者修改标题/封面后可重新进入初始池。</p>
<h3>1.4 运营与算法的协同</h3>
<p>运营通过&quot;调控因子&quot;影响推荐结果，而非绕过推荐：</p>
<table>
<thead>
<tr>
<th>调控手段</th>
<th>实现方式</th>
<th>示例</th>
</tr>
</thead>
<tbody><tr>
<td><strong>Boost / Demote</strong></td>
<td>排序分加减权重</td>
<td>&quot;世界杯相关内容 Boost 2x&quot;</td>
</tr>
<tr>
<td><strong>流量预留</strong></td>
<td>预留固定比例位置给运营内容</td>
<td>第 3 位固定为运营推荐位</td>
</tr>
<tr>

<td>&quot;美食 ≤ 30%，科技 ≥ 10%&quot;</td>
</tr>
<tr>
<td><strong>时效性调控</strong></td>

</tr>

<p>这些调控因子通过配置中心下发，在重排阶段读取并应用。<strong>好处是运营干预可追溯、可回滚、可量化效果。</strong></p>
<p><strong>一句话总结</strong>：运营不是和算法对着干，而是给算法&quot;调参&quot;——用业务判断弥补模型的盲区。</p>

<h3>3.3 隐私保护的架构视角</h3>
<p>合规不只是一份 checklist，它对推荐系统的架构设计有实质影响。</p>
<p><strong>联邦学习在推荐中的应用</strong>：传统方案将用户行为数据集中到服务端训练模型，联邦学习的思路是让模型到数据所在地训练，只上传梯度而非原始数据。Google 的 Federated Learning of Cohorts（FLoC，已演进为 Topics API）是这个方向的工业级尝试。对于内容社区，联邦学习的落地场景是<strong>端侧行为建模</strong>——在用户设备上训练轻量的兴趣模型，只将兴趣标签（而非完整行为序列）上传到服务端参与召回。</p>
<p><strong>差分隐私对推荐效果的影响</strong>：在训练数据中加入噪声是差分隐私的核心思路，但噪声会降低模型精度。实验表明，在 ε=10 的差分隐私约束下，推荐模型的 CTR 预估 AUC 通常下降 1-3%。这个精度损失在大规模数据（千万级样本）下可以接受，但在冷启动或小众类目上会被放大。<strong>架构设计上的平衡</strong>：对高频行为（点击、播放）加较小噪声，对敏感行为（搜索、位置）加较大噪声，分级保护。</p>
<p><strong>端智能与隐私保护的协同</strong>：上篇提到的短视频预加载、实时重排等端侧能力，天然具有隐私优势——敏感行为数据不出端。将部分推理下放到客户端，既降低了服务端的隐私风险，又减少了网络延迟。这是隐私合规和用户体验的双赢路径。</p>

<h3>4.1 LLM 在推荐系统中的新场景</h3>
<p>上篇第 4.1 节已经讨论了 LLM 驱动的内容理解（多任务推理替代多个专用模型）。LLM 在推荐系统中的应用正在向更多场景扩展：</p>
<p><strong>推荐理由生成</strong>：给每条推荐内容附上一句自然语言解释——&quot;因为你昨天看了露营攻略&quot;。传统方案用规则模板生成（如&quot;XXX 也在看&quot;），LLM 可以生成更自然、更个性化的推荐理由，实验数据显示 CTR 提升 3-8%。</p>
<p><strong>对话式搜索</strong>：传统搜索是&quot;输入关键词→返回列表&quot;的单轮模式。LLM 使多轮对话式搜索成为可能——用户说&quot;找一个适合新手的露营地&quot;，系统追问&quot;你在哪个城市？偏好山区还是海边？&quot;，逐步缩小范围。小红书已在部分场景上线了类似能力。</p>
<p><strong>创作者 Copilot</strong>：用 LLM 辅助创作者生成标题候选、优化封面文案、分析同类爆款的共性。这既提升了内容供给质量，也增加了创作者对平台的粘性。</p>

agent.run(&quot;&quot;&quot;
请审核以下内容：

&quot;&quot;&quot;)
</code></pre>

<h2>总结</h2>
<p>信息流架构不是一个单点系统，而是一条<strong>端到端的数据流水线</strong>。它的质量取决于每个环节的协同效率，而非某个组件有多厉害。</p>
<p>四条核心原则：</p>
<ol>
<li><strong>业务和技术双轮驱动</strong>。纯靠算法做不好社区（模型不理解业务意图），纯靠运营也扛不住规模（人工无法处理百万级内容）。两者通过&quot;调控因子&quot;实现协同</li>
<li><strong>渐进式架构演进</strong>。不要在 MVP 阶段就搭建抖音级别的推荐系统。从规则推荐开始，每一次升级都由明确的业务瓶颈驱动</li>
<li><strong>度量驱动优化</strong>。没有 AB 实验和全链路 Trace，就不要谈推荐优化。先建度量体系，再谈模型迭代</li>
<li><strong>创新选对环节</strong>。LLM 用在内容理解和交互、在线学习用在排序、端智能用在体验——在正确的环节引入正确的技术，才是真正的技术判断力</li>
</ol>

<h2>1. 为什么需要平台</h2>
<p>大多数 Agent 教程的终点是一个能跑通的 demo。从 demo 到生产、从一个 Agent 到一百个 Agent，中间隔着的不是&quot;更多代码&quot;——是基础架构平台。</p>
<p>类比早期互联网公司后端：每个团队各自搭 Web 框架、各自管数据库连接、各自实现日志系统。后来演化出统一的微服务平台——服务注册发现、配置中心、API 网关、链路追踪、容器编排——开发者只关注业务逻辑。</p>
<p>Agent 领域正在经历同样的演进。当前许多团队每个 Agent 项目自己管 LLM 调用、自己实现会话存储、自己处理工具调用、自己搭监控——<strong>大量重复建设，质量参差不齐</strong>。</p>
<p>平台演进三阶段：</p>

<h2>2. 与传统微服务平台的四个本质差异</h2>
<p>理解这四个差异，才能理解后续每层设计中那些&quot;看似过度&quot;的部分——它们都在应对 Agent 场景的本质挑战。</p>
<h3>2.1 长会话 vs 短请求</h3>
<p>传统微服务处理无状态短请求——毫秒到秒级，请求间无关联。Agent 交互是长会话——一次会话可能数十轮对话，持续几分钟到几小时，每轮决策依赖之前所有轮次。</p>
<p><strong>工程后果</strong>：会话状态必须持久化；负载均衡需要 Session 亲和性；资源分配粒度从&quot;请求级&quot;变为&quot;会话级&quot;。</p>
<h3>2.2 非确定性 vs 确定性</h3>
<p>传统服务执行路径确定——代码审查就能推断行为。Agent 核心是 LLM 推理——同一输入可能产生不同的工具调用序列、不同的执行路径。<strong>一次请求可能走 2 轮也可能走 8 轮</strong>。</p>
<p><strong>工程后果</strong>：资源消耗不可预测，需要更灵活的超时和限流；排障不能只靠日志，需要完整运行时追踪；质量保障不能只靠单元测试，需要统计性评估体系。</p>
<h3>2.3 Token 经济 vs 计算经济</h3>
<p>传统服务的主要成本是 CPU 和内存——可以通过加机器线性扩展。Agent 的主要成本是 Token——<strong>旗舰模型与轻量模型可相差 10-30 倍</strong>。</p>
<p><strong>工程后果</strong>：流控核心维度从 QPS 变为 TPM（Tokens Per Minute）；成本优化是省 Token，不是省机器——Prompt 压缩、模型路由、缓存命中率成为关键指标；预算管理需精确到单次调用级别。</p>
<h3>2.4 工具调用的安全边界</h3>
<p>传统服务调下游 API 是开发者编码时确定的——调什么、传什么写死。Agent 的工具调用是 LLM 运行时动态决策的——<strong>调哪个工具、传什么参数取决于 LLM 推理结果</strong>。</p>
<p><strong>工程后果</strong>：工具调用需要沙箱隔离；参数需要运行时校验，不能信任 LLM 输出；敏感操作需要人工审批机制。</p>

<h3>4.3 Token 维度的流控</h3>
<p><strong>这是 Agent 平台接入层最独特的设计</strong>。传统流控以 QPS 为核心——一个请求的资源消耗大致相同。</p>
<p>Agent 平台中<strong>同一个请求的资源消耗可能相差 100 倍</strong>——简单问答 500 Token，复杂多步推理 50000 Token。仅限制 QPS 无法保护模型资源——10 个&quot;重请求&quot;可能比 1000 个&quot;轻请求&quot;消耗更多预算。</p>
<p>双维度流控：QPS 保护接入层和调度层，TPM 保护模型资源和预算。</p>
<p><strong>TPM 流控的难点是请求开始时 Token 消耗未知</strong>。实用方案是<strong>预估 + 事后结算</strong>——类似信用卡的授权和结算：</p>
<pre><code class="language-python">def admit_request(req: Request, tenant: Tenant) -&gt; AdmissionResult:
    &quot;&quot;&quot;请求进入时：预估 token 消耗 + 预扣额度&quot;&quot;&quot;
    # 1. 估算这次请求大约多少 token
    estimated = estimate_token_consumption(
        input_tokens=count_tokens(req.input),
        agent_kind=req.agent_kind,
    )
    # 简单 Agent ×2 系数，复杂 Agent ×5 系数
    multiplier = AGENT_MULTIPLIER.get(req.agent_kind, 3)
    estimated_total = estimated * multiplier

    # 2. 检查租户 TPM 余额（用 Redis + Lua 保证原子性）
    ok = redis_eval(LUA_RESERVE_TOKENS, keys=[tenant.tpm_key],
                    args=[estimated_total, now_minute()])
    if not ok:
        return AdmissionResult(admit=False, reason=&quot;tpm_limit&quot;, retry_after_s=30)

    # 3. 预扣，记录预扣量到 reservation_table
    reservation_id = reserve(tenant.id, req.id, estimated_total)
    return AdmissionResult(admit=True, reservation_id=reservation_id,
                           estimated_tokens=estimated_total)

def settle_request(reservation_id: str, actual_tokens: int):
    &quot;&quot;&quot;请求结束时：用实际消耗替换预扣量&quot;&quot;&quot;
    r = get_reservation(reservation_id)
    diff = actual_tokens - r.estimated

    # 实际比预估多/少都要更新租户余额
    redis_incrby(r.tenant.tpm_key, diff)
    record_actual(r.tenant.id, r.request_id, actual_tokens)

    # 用实际数据动态调整该 Agent 的预估系数
    update_multiplier(r.agent_kind,
                     ratio=actual_tokens / r.estimated)
</code></pre>

<th>动作</th>

<td>根据输入长度和 Agent 类型估算（简单 Agent 2×，复杂 5×）→ 检查租户 TPM 余额 → 预扣预估量</td>
</tr>

<h2>5. 调度层：Agent Registry 与多租户</h2>
<p>调度层让平台从&quot;跑一个 Agent&quot;跨越到&quot;管一群 Agent&quot;。</p>
<h3>5.1 Agent Registry</h3>
<p>类似微服务注册中心（Nacos、Consul），但管理的元数据不同——Agent 特有的属性包括：</p>
<pre><code class="language-yaml">agent:
  id: &quot;agent-customer-support-v2&quot;
  version: &quot;2.3.1&quot;
  status: &quot;active&quot;  # active / canary / deprecated / disabled

  config:
    model: &quot;gpt-4o&quot;
    max_turns: 10
    system_prompt_ref: &quot;prompt://customer-support/v2.3&quot;
    tools: [&quot;tool://order-query&quot;, &quot;tool://refund-process&quot;]

  routing:
    tenants: [&quot;tenant-a&quot;, &quot;tenant-b&quot;]
    traffic_weight: 80          # 灰度流量百分比
    fallback_agent: &quot;agent-customer-support-v1&quot;

  health:
    active_sessions: 47
    avg_latency_p95: 3200ms
</code></pre>
<p>存储选型按规模：&lt; 100 个 Agent 用 Redis/etcd；100-1000 用 PostgreSQL + 缓存；&gt; 1000 用专门的元数据服务。</p>
<p><strong>配置更新与热生效</strong>对 Agent 平台特别重要——Prompt 修改、模型切换、工具调整，都需要在不重启的情况下立即生效。</p>

<td><strong>数据隔离</strong></td>
<td>共享存储 + tenant_id 过滤（逻辑），或独立数据库（物理）。安全要求高的租户用物理</td>
</tr>
<tr>
<td><strong>模型隔离</strong></td>
<td>不同租户的请求队列独立——某租户占满模型供应商速率限制不应影响其他租户</td>
</tr>
</tbody></table>
<p>多层配额检查的代码骨架：</p>
<pre><code class="language-python">def check_quotas(req: Request, tenant: Tenant, agent: Agent, session: Session):
    &quot;&quot;&quot;每次进入 LLM 调用前要过 4 层配额检查&quot;&quot;&quot;
    checks = [
        # 平台级
        (&quot;platform_tpm&quot;,
         platform_tpm_used() &lt; PLATFORM_TPM_LIMIT,
         &quot;platform-wide tpm exhausted&quot;),
        # 租户级
        (&quot;tenant_monthly&quot;,
         tenant.monthly_used &lt; tenant.monthly_quota,
         &quot;tenant monthly token quota exceeded&quot;),
        (&quot;tenant_concurrency&quot;,
         tenant.active_sessions &lt; tenant.max_sessions,
         &quot;tenant concurrent sessions limit&quot;),
        # Agent 级
        (&quot;agent_per_minute&quot;,
         agent.calls_this_minute &lt; agent.max_per_minute,
         &quot;agent per-minute call limit&quot;),
        # 会话级
        (&quot;session_token&quot;,
         session.tokens &lt; session.max_tokens,
         &quot;session token limit reached&quot;),
    ]
    for name, ok, msg in checks:
        if not ok:
            raise QuotaExceeded(scope=name, message=msg)
</code></pre>

<h3>7.1 LLM Gateway</h3>
<p>能力层最核心的组件——在 Agent 运行时和 LLM 供应商之间建立抽象层。</p>
<p><img src="/images/blog/agentic/llm-gateway.svg" alt="LLM Gateway 架构"></p>
<table>
<thead>
<tr>
<th>能力</th>
<th>实现</th>
</tr>
</thead>
<tbody><tr>
<td><strong>统一接口</strong></td>
<td>屏蔽不同供应商（OpenAI/Anthropic/Google/本地）的 API 差异，对上提供统一调用接口</td>
</tr>
<tr>
<td><strong>智能路由</strong></td>
<td>根据请求特征（任务复杂度、Token 长度、延迟要求、成本预算）选择模型——简单问答路由到 GPT-4o-mini，复杂推理路由到 Claude Opus</td>
</tr>
<tr>
<td><strong>Fallback</strong></td>
<td>主模型超时或不可用时降级到备选。Fallback 链：主模型 → 同级备选 → 低一级 → 错误响应</td>

<tr>
<td><strong>语义缓存</strong></td>
<td>相同或语义相似的请求直接返回缓存。基于 Embedding 相似度匹配，命中率和准确率需持续监控</td>

<tr>

<td>模型供应商速率限制成为瓶颈时，按租户优先级调度——高优先级先发，低优先级排队</td>
</tr>
</tbody></table>
<p>LLM Gateway 的核心调用伪代码：</p>
<pre><code class="language-python">class LLMGateway:
    def chat(self, req: ChatRequest, ctx: RequestContext) -&gt; ChatResponse:
        # 1. 语义缓存查找（仅对幂等、可缓存请求）
        if req.cacheable:
            if cached := semantic_cache.get(req, threshold=0.93):
                return cached

        # 2. 路由：根据请求特征 + 当前供应商状态选择模型
        model = router.select(
            features=extract_features(req),
            constraints={
                &quot;max_cost&quot;: ctx.budget_remaining,
                &quot;max_latency_p95&quot;: ctx.latency_sla,
                &quot;tenant_tier&quot;: ctx.tenant.tier,
            },
        )

        # 3. 调用，带 fallback 链
        for attempt, candidate in enumerate(model.fallback_chain):
            try:
                resp = provider_call(
                    candidate, req,
                    timeout=ctx.timeout_budget(attempt),
                )
                if req.cacheable and resp.is_high_quality:
                    semantic_cache.set(req, resp)
                return resp
            except (TimeoutError, RateLimitError, ProviderError) as e:
                log_fallback_event(ctx, candidate, e)
                if attempt == len(model.fallback_chain) - 1:
                    raise          # 链路用尽
        raise UnreachableException()
</code></pre>
<p><strong>语义缓存的陷阱</strong>——它在某几类请求上是金矿（基础知识问答、固定流程的客服 FAQ、确定性的工具用法指导），在另几类上是地雷：</p>
<table>
<thead>
<tr>
<th>不能缓存的场景</th>
<th>为什么</th>
<th>失败时的代价</th>
</tr>
</thead>
<tbody><tr>
<td><strong>实时数据查询</strong>（今天股价、当前订单状态）</td>
<td>缓存命中时返回的是过时数据</td>
<td>业务决策基于陈旧信息</td>
</tr>
<tr>
<td><strong>个性化响应</strong>（&quot;我的订单&quot;、&quot;我的余额&quot;）</td>
<td>不同用户问&quot;我的 X&quot;语义相似，命中错误用户的缓存</td>
<td><strong>跨用户数据泄露</strong></td>
</tr>
<tr>
<td><strong>带状态的操作</strong>（&quot;刚才那个改成...&quot;）</td>
<td>同样的提问在不同对话上下文里答案完全不同</td>
<td>答非所问</td>
</tr>
<tr>
<td><strong>创意/写作</strong>（&quot;写个诗&quot;、&quot;想个名字&quot;）</td>
<td>用户预期每次都新鲜</td>
<td>用户体验断崖</td>
</tr>
<tr>
<td><strong>基于工具结果的回答</strong></td>
<td>工具结果本身可能变了</td>
<td>给出与工具新结果不一致的答案</td>
</tr>
</tbody></table>
<p>工程上的两条保险：<strong>命中后必须验证关键字段没漂移</strong>（如缓存的回答里说&quot;截至 X 时&quot;，X 时如果已经过期就跳过缓存）；<strong>对高风险路径默认禁用缓存</strong>（涉及金额、账户、合规决策的请求强制走全流程）。<strong>缓存错误的回答比不缓存更危险</strong>——一次错误回答只是体验问题，缓存它会让 N 倍用户复制同一错误。</p>

<td>本地模型加载几十秒到几分钟——预加载常用模型，保持&quot;热实例&quot;随时可用</td>
</tr>
</tbody></table>
<p>GPU 调度的几个工程现实，比&quot;分时复用、负载感知、冷启动&quot;这三行字更扎心：</p>
<ul>
<li><strong>碎片化代价高</strong>：A100 80GB 拆 4 张 20GB 跑 7B 模型看起来划算，但每张 vGPU 的 KV cache 上限随之降为 1/4，并发 token 容量同步缩水——不是简单地&quot;算力÷4&quot;。MIG 模式比 MPS 隔离更强但灵活度更低，团队要按模型混部模式选</li>
<li><strong>同租户优先</strong>：跨租户共享 GPU 时，QoS 抖动会让两个租户都不满意。除非负载特征互补（一个白天热、一个晚上热），否则物理隔离比逻辑隔离运维更省心</li>
<li><strong>模型预热是真活</strong>：vLLM/SGLang 加载 70B 模型需要约 1-3 分钟，期间所有请求要排队。生产中通常按&quot;工作时段&quot;预热常用模型，深夜降级到几个保留实例就够</li>
<li><strong>混合部署的拐点</strong>：当某个模型的日均 token 量 ≥ 1B token/月时，本地部署的固定成本（GPU 折旧 + 运维）通常已经低于 API 费用——这个阈值是判断&quot;什么时候该从纯 API 切到混合部署&quot;的实用经验值</li>
</ul>
<h3>8.4 成本工程：平台层的杠杆</h3>
<p>单 Agent 视角的成本优化——模型分层、缓存、Prompt 压缩、工具结果截断、Spot Instance——这是每个 Agent 项目都要做的事。平台层多出两个杠杆：</p>
<table>

<th>节省</th>
<th>平台层独有的理由</th>
</tr>
</thead>

<td>30-60%</td>
<td>Agent 之间可共享缓存——租户 A 查过的问题，租户 B 也可能问到（在不违反数据隔离的前提下）</td>

<h2>10. 安全治理：平台层的统一拦截</h2>

<h2>12. 不是微服务复刻，是一组新约束下的重新设计</h2>
<p>Agent 基础架构平台容易被低估，因为表面看它和微服务平台很像——都有网关、注册、调度、可观测。但 Token 是新货币（成本以 token 计、不是以 CPU 时间计）、Session 是新单位（一次&quot;请求&quot;是一次几分钟的循环、不是一次毫秒级 API 调用）、非确定性是新常态（同一输入路径不同、资源消耗不同）——这三个本质差异让平台的每一层都需要重新设计，而不是把微服务平台的模式照搬过来。</p>
<p>平台层和单 Agent 层解决的是不同的问题。单 Agent 层关心&quot;这一次 Agent 怎么把任务做对&quot;，平台层关心&quot;几十上百个 Agent 跑在同一套基础设施上怎么不互相干扰、不重复造轮子、不让成本和故障互相传染&quot;。前者的工程焦点是 Memory、Planner、Tools；后者的工程焦点是 Token 配额、租户隔离、共享能力、统一观测。两层都做好才有规模化 Agent 系统。</p>
<p>平台演进的最大风险是 Phase 0 就规划 Phase 3 的系统。从最小可用开始——LLM Gateway 一个、日志一套、Agent 一个，让真实运营痛点驱动架构演进。先有第一个 Agent 上生产，再谈多 Agent 管理，再谈成本治理。这是规避过度设计的唯一方法，也是贯穿整个 Agentic 工程的统一哲学：用最简单的抽象解决问题，复杂度只在必要时引入。</p>
18:Td4ff,<blockquote>
<p>信息流是内容社区的中枢神经——它决定了创作者的内容能被谁看到，用户打开 App 会看到什么，平台的商业价值如何兑现。架构设计的每一个决策，最终都会反映在用户体验和业务指标上。</p>
</blockquote>
<p>本文是信息流系列的上篇，聚焦<strong>技术架构</strong>——从内容生产 Pipeline、特征工程、召回排序引擎，到 Feed 分发、搜索、互动、审核、AB 实验，完整拆解&quot;信息流系统怎么建&quot;。目标读者是后端工程师、算法工程师和架构师。如果你更关心运营体系、商业化路径和团队建设，请阅读下篇<a href="/blog/engineering/architecture/%E7%99%BE%E4%B8%87%E7%BA%A7%E5%86%85%E5%AE%B9%E7%A4%BE%E5%8C%BA%E4%BF%A1%E6%81%AF%E6%B5%81%EF%BC%88%E4%B8%8B%EF%BC%89%EF%BC%9A%E8%BF%90%E8%90%A5%E4%BD%93%E7%B3%BB%E4%B8%8E%E5%95%86%E4%B8%9A%E5%8C%96">《百万级内容社区信息流（下）：运营体系与商业化》</a>。</p>
<p>文章的展开顺序和数据流一致：先理解问题本质和全景架构，然后沿着&quot;内容从创建到被用户看到&quot;的链路逐层深入——生产、理解、召回排序、分发，再延伸到搜索、互动、审核、度量四个支撑子系统。</p>

<h2>1. 内容社区的核心问题</h2>
<p>内容社区的本质是一个<strong>三方博弈系统</strong>：创作者生产内容、用户消费内容、平台在中间做分配和治理。</p>
<p>这三方的利益并不天然一致：</p>
<table>
<thead>
<tr>
<th>角色</th>
<th>核心诉求</th>
<th>典型矛盾</th>
</tr>
</thead>
<tbody><tr>
<td><strong>创作者</strong></td>
<td>获得曝光、互动、收益</td>
<td>优质内容未必获得流量，标题党反而有效</td>

<tr>
<td><strong>用户</strong></td>
<td>看到感兴趣的、高质量的内容</td>

<tr>
<td><strong>平台</strong></td>

<td>过度商业化伤害体验，过度管控抑制供给</td>
</tr>
</tbody></table>
<p><strong>信息流是连接三方的核心枢纽——它决定了&quot;谁看到什么&quot;。</strong> 推荐做得好，三方共赢；做得差，劣币驱逐良币，生态崩塌。</p>
<p>从百万级用户向千万级迈进的过程中，信息流架构会遭遇三个关键转折点：</p>
<ol>
<li><strong>内容量爆炸</strong>：日均新增内容从千级到十万级，规则推荐失效，必须引入模型</li>
<li><strong>兴趣分化</strong>：用户群体从同质化到多元化，&quot;千人一面&quot;变成&quot;千人千面&quot;</li>
<li><strong>生态治理</strong>：低质内容、水军、搬运、擦边开始涌现，审核和调控成为刚需</li>
</ol>
<p>这三个转折点，分别对应架构上的三次升级：<strong>从规则到模型、从离线到实时、从单链路到全链路治理。</strong></p>

<td>效果度量、内容治理、合规保障</td>
<td>AB 实验指标、推荐 Trace、审核风控、隐私合规</td>
<td>近实时</td>
</tr>
<tr>
<td><strong>基础设施层</strong></td>
<td>存储、计算、网络</td>
<td>Kafka、Redis、MySQL/TiDB、ES、Milvus、Flink</td>
<td>—</td>
</tr>
</tbody></table>
<p>需要区分两个&quot;200ms&quot;：<strong>召回排序层的 200ms 是模型链路延迟</strong>（从收到请求到排序完成），<strong>分发渲染层的 P99 &lt; 300ms 是端到端延迟</strong>（包含模型链路 + Feed 服务逻辑 + 序列化 + 网络传输）。后文会分别讨论。</p>
<p><strong>核心设计原则</strong>：</p>
<ul>
<li><strong>读写分离</strong>：内容写入链路（发布→审核→入库）和读取链路（请求→召回→排序→返回）完全解耦，写入慢不影响读取体验</li>
<li><strong>三级流水线</strong>：离线（T+1 批量计算）、近线（Flink 分钟级流处理）、在线（请求级实时计算）各司其职</li>
<li><strong>内容与分发解耦</strong>：内容的存储和理解独立于分发策略，同一份内容可以出现在推荐流、关注流、搜索结果等多个场景</li>
</ul>
<p><strong>一句话总结</strong>：信息流架构的本质是一条数据流水线——内容从创建到被用户看到，经过层层处理和筛选，每一层都在做&quot;减法&quot;和&quot;排序&quot;。</p>

<h3>3.3 内容元数据模型</h3>
<p>每条内容入库时，会生成一套结构化的元数据，这是后续召回和排序的基础：</p>
<pre><code class="language-python">@dataclass
class ContentMeta:
    content_id: str
    content_type: str           # &quot;article&quot; | &quot;short_video&quot; | &quot;image_post&quot;
    author_id: str

    # 内容理解结果

    sub_category: str           # 二级类目

<p><strong>数据质量直接决定推荐效果——垃圾进、垃圾出。</strong> 内容理解的准确率每提升 1%，下游推荐的相关性指标通常能提升 0.3-0.5%。</p>
<p><strong>一句话总结</strong>：内容 Pipeline 的核心使命是&quot;又快又准&quot;——快速完成审核和处理，为下游推荐提供高质量的原材料。</p>

<p><strong>一句话总结</strong>：特征工程的核心挑战不是&quot;提取什么特征&quot;，而是&quot;如何在毫秒级延迟内把正确版本的特征送到模型面前&quot;。</p>

<h3>5.3 内容侧冷启动</h3>
<p>新内容没有任何互动数据，是召回和排序的&quot;盲区&quot;。如果冷启动做不好，新内容得不到曝光，创作者流失，供给端萎缩——这是百万级社区最常遇到的实际问题。</p>
<p><strong>冷启动的核心思路是 Exploration-Exploitation 平衡</strong>：既要给新内容试探机会（Exploration），又不能让大量低质新内容冲击用户体验（Exploitation）。</p>
<p><strong>初始分数的来源</strong>：新内容没有互动数据，但有内容理解产出的特征——质量分、原创度、美学评分、作者历史表现。用这些特征构建一个<strong>冷启动评分模型</strong>，为新内容分配初始排序分。</p>
<table>
<thead>
<tr>
<th>信号</th>
<th>权重</th>
<th>说明</th>
</tr>
</thead>
<tbody><tr>
<td><strong>内容质量分</strong></td>
<td>高</td>
<td>来自 3.3 节的 quality_score</td>
</tr>
<tr>
<td><strong>作者历史表现</strong></td>
<td>高</td>
<td>作者过往内容的平均 CTR、互动率</td>
</tr>
<tr>
<td><strong>原创度</strong></td>
<td>中</td>
<td>搬运内容初始分更低</td>
</tr>
<tr>
<td><strong>内容完整度</strong></td>
<td>中</td>
<td>图文长度、视频时长是否合理</td>
</tr>
<tr>
<td><strong>美学评分</strong></td>
<td>低</td>
<td>封面/配图质量</td>
</tr>
</tbody></table>
<p><strong>流量池赛马机制</strong>（抖音模式）：新内容进入分级流量池，每一级通过数据表现决定是否晋级：</p>
<table>
<thead>
<tr>

<th>淘汰后果</th>
</tr>
</thead>
<tbody><tr>
<td><strong>初始池</strong></td>
<td>300-500</td>
<td>2-4 小时</td>
<td>完播率 &gt; 15%，互动率 &gt; 3%</td>
<td>停止分发，沉入长尾</td>

<td>6-12 小时</td>

<p><strong>工程实现要点</strong>：流量池的分级不是物理隔离，而是在重排阶段通过 Boost 因子控制。新内容在初始池阶段获得一个正向 Boost，数据达标后 Boost 系数增大（进入下一级池），未达标则 Boost 归零。这个机制可以和 AB 实验平台联动——不同的晋升阈值作为实验参数调优。</p>
<p><strong>一句话总结</strong>：冷启动不是&quot;给新内容更多流量&quot;那么简单，而是一套&quot;小范围试探→数据验证→逐级放大&quot;的风控机制。</p>
<h3>5.4 粗排</h3>
<p>粗排的目标是&quot;快速筛选&quot;——用轻量模型从万级候选中选出千级。这是<strong>性能瓶颈最敏感的环节</strong>——候选量大（万级），但延迟预算只有 30ms。</p>
<p><strong>双塔模型</strong>是粗排的标准方案：用户塔和内容塔分别生成 Embedding，用内积或余弦相似度打分。</p>
<p><strong>为什么双塔适合粗排</strong>：</p>
<table>
<thead>
<tr>
<th>设计选择</th>
<th>原因</th>
</tr>
</thead>
<tbody><tr>
<td>用户塔和内容塔独立计算</td>
<td>用户 Embedding 可缓存，同一用户短时间内不需要重复计算</td>
</tr>
<tr>
<td>只用静态特征</td>
<td>粗排不加入实时交叉特征，避免实时计算开销</td>
</tr>
<tr>
<td>内积打分</td>
<td>简单向量运算，万级候选打分 &lt; 5ms</td>
</tr>
</tbody></table>
<p><strong>粗排的工程优化</strong>：</p>
<table>
<thead>
<tr>
<th>优化手段</th>
<th>效果</th>
</tr>
</thead>
<tbody><tr>
<td><strong>用户 Embedding 缓存</strong></td>
<td>缓存周期 10-30 分钟，避免每次请求都过用户塔</td>
</tr>
<tr>
<td><strong>批量推理</strong></td>
<td>万级候选一次性批量打分，GPU 利用率更高</td>
</tr>
<tr>
<td><strong>量化压缩</strong></td>
<td>Embedding 从 FP32 量化到 INT8，内存减半，计算加速 2-3x</td>
</tr>
<tr>
<td><strong>异步预计算</strong></td>
<td>新内容入库时提前过内容塔，存储 Embedding，请求时直接读取</td>
</tr>
</tbody></table>
<p><strong>粗排和精排的核心差异</strong>：</p>
<table>
<thead>
<tr>
<th>维度</th>
<th>粗排</th>
<th>精排</th>
</tr>
</thead>
<tbody><tr>
<td>候选量</td>
<td>万级</td>
<td>千级</td>
</tr>
<tr>
<td>特征</td>
<td>静态特征（Embedding、类目、标签）</td>
<td>静态 + 实时特征（行为序列、上下文）</td>
</tr>
<tr>
<td>模型</td>
<td>双塔，参数量小</td>
<td>DIN/DIEN/多目标 MTL，参数量大</td>
</tr>
<tr>
<td>推理方式</td>
<td>向量内积，批量打分</td>
<td>逐条精细打分</td>
</tr>
<tr>
<td>延迟</td>
<td>&lt; 30ms</td>
<td>&lt; 80ms</td>
</tr>
</tbody></table>
<p><strong>训练数据差异</strong>：粗排用全量曝光数据训练（正负样本充足），精排用粗排后的数据训练（更接近真实分布）。一个常见错误是用精排的训练数据训练粗排——这会导致粗排&quot;只认识精排筛过的内容&quot;，丢失多样性。</p>
<h3>5.5 精排</h3>
<p>精排是整个推荐链路中最&quot;重&quot;的环节——它直接决定用户看到的内容顺序。</p>
<p><strong>模型演进路径</strong>：</p>
<table>
<thead>
<tr>
<th>阶段</th>
<th>模型</th>
<th>特点</th>
<th>代表产品</th>
</tr>
</thead>
<tbody><tr>
<td>早期</td>
<td>LR / GBDT</td>
<td>手工特征，可解释</td>
<td>早期微博</td>
</tr>
<tr>
<td>中期</td>
<td>Wide &amp; Deep / DeepFM</td>
<td>自动特征交叉</td>
<td>小红书早期</td>
</tr>
<tr>
<td>当前主流</td>
<td>DIN / DIEN / SIM</td>
<td>捕捉用户行为序列中的兴趣变化</td>
<td>抖音、B站</td>
</tr>
<tr>
<td>前沿</td>
<td>多目标 MTL + 在线学习</td>
<td>同时优化多个目标，实时更新</td>
<td>抖音推荐</td>
</tr>
</tbody></table>
<p><strong>多目标优化</strong>是当前精排的核心技术点。单目标优化 CTR 会导致&quot;标题党泛滥&quot;，单目标优化完播率会导致&quot;信息茧房&quot;。</p>
<p>实际的优化目标是多个指标的加权组合：</p>
<pre><code class="language-python"># 多目标融合公式（简化版）
final_score = (
    w_ctr * predict_ctr          # 点击率
    + w_finish * predict_finish   # 完播率（视频）/ 完读率（图文）
    + w_like * predict_like       # 点赞概率
    + w_comment * predict_comment # 评论概率
    + w_share * predict_share     # 分享概率
    + w_follow * predict_follow   # 关注概率
    - w_dislike * predict_dislike # 负反馈概率
)
</code></pre>
<p>权重 <code>w_*</code> 的调整直接影响社区生态走向。抖音团队在公开分享中提到，他们根据业务阶段动态调整权重——拉新期加大 CTR 权重促进增长，成熟期加大完播和互动权重提升留存。</p>
<p><strong>短视频精排的特殊性</strong>：短视频流（类似抖音的沉浸式上下滑）和图文信息流在排序目标上有关键差异。图文流的核心指标是 CTR（点击率），用户通过标题和封面决策是否点击，决策时间 1-2 秒；短视频流的核心指标是完播率和播放时长，用户看前 3 秒决定是否继续，完播是非常强的正向信号，3 秒内划走是强烈的负信号。此外，短视频有独特的召回维度——很多热门视频共享同一首 BGM 或同一个模板，BGM/模板 ID 是有效的 Item-CF 信号。多模态理解也更难，需要同时处理画面、声音、字幕、BGM 四个通道。</p>
<h3>5.6 重排</h3>
<p>重排是排序链路的最后一关，负责&quot;生态调控&quot;——在精排结果的基础上做全局优化。</p>
<table>
<thead>
<tr>
<th>重排策略</th>
<th>目的</th>
<th>实现方式</th>
</tr>
</thead>
<tbody><tr>
<td><strong>多样性打散</strong></td>
<td>避免连续出现同类内容</td>
<td>滑动窗口内同类目 / 同作者限频</td>
</tr>
<tr>
<td><strong>频控去重</strong></td>
<td>避免用户重复看到相同内容</td>
<td>布隆过滤器记录已曝光内容</td>
</tr>
<tr>
<td><strong>新内容加权</strong></td>
<td>保证新内容有曝光机会</td>
<td>发布 N 小时内的内容给予 Boost</td>
</tr>
<tr>
<td><strong>创作者保底</strong></td>
<td>保证活跃创作者的内容有基础流量</td>
<td>每个创作者每天至少 N 次曝光</td>
</tr>
<tr>
<td><strong>运营置顶</strong></td>
<td>热点事件、活动推广</td>
<td>运营后台配置干预位</td>
</tr>
<tr>
<td><strong>广告混排</strong></td>
<td>信息流广告插入</td>
<td>固定位置或动态竞价</td>
</tr>
</tbody></table>
<p>小红书的重排策略特别强调<strong>多样性</strong>——同一屏内不会出现两条同品类的笔记，这是其&quot;逛街感&quot;体验的技术保障。</p>
<p><strong>一句话总结</strong>：召回负责&quot;不漏&quot;，粗排负责&quot;快筛&quot;，精排负责&quot;排准&quot;，重排负责&quot;调生态&quot;。四个阶段各有分工，缺一不可。</p>

<h2>6. Feed 分发与高可用</h2>
<p>Feed 服务是用户端直接感知的入口——用户下拉刷新看到的每一条内容，都是 Feed 服务返回的。</p>
<h3>6.1 Push vs Pull vs 混合</h3>
<p><img src="/images/blog/community-feed-architecture/04-feed-distribution.svg" alt="Feed 分发架构"></p>
<table>
<thead>
<tr>
<th>模式</th>
<th>原理</th>
<th>优势</th>
<th>劣势</th>
<th>适用场景</th>
</tr>
</thead>
<tbody><tr>
<td><strong>Push</strong></td>
<td>创作者发布时，写入所有粉丝的 Inbox</td>
<td>读取极快</td>
<td>大 V 一条写入百万次</td>
<td>关注流 + 粉丝 &lt; 1 万</td>
</tr>
<tr>
<td><strong>Pull</strong></td>
<td>用户请求时，实时拉取并排序</td>
<td>写入零成本</td>
<td>读取延迟高</td>
<td>推荐流、发现页</td>
</tr>
<tr>
<td><strong>混合</strong></td>
<td>大 V 用 Pull，小 V 用 Push</td>
<td>平衡读写</td>
<td>实现复杂</td>
<td>粉丝 &gt; 10 万</td>
</tr>
</tbody></table>
<p><strong>百万级用户下的选型</strong>：</p>
<ul>
<li><strong>关注流</strong>：Push 模式。粉丝量 &lt; 1 万的创作者发布时直接写入粉丝 Inbox（Redis Sorted Set）。粉丝量 &gt; 10 万的走混合模式</li>
<li><strong>推荐流</strong>：Pull 模式。每次请求触发实时召回和排序</li>
<li><strong>热点/同城流</strong>：Pull + 缓存。候选列表分钟级刷新，用户请求时个性化重排</li>
</ul>
<p>微博是 Push 模式的经典案例——曾因大 V 发微博导致写入风暴而多次宕机，后来改为混合模式。</p>
<h3>6.2 分页与去重</h3>
<p><strong>分页策略</strong>：不能用传统的 offset/limit（内容实时变化会导致重复或跳过）。推荐流用 <strong>cursor-based 分页</strong>——每次返回一个不透明的 cursor token，包含上次请求的状态。</p>
<p><strong>去重策略</strong>：服务端用<strong>布隆过滤器</strong>记录每个用户近期已曝光的内容 ID（过期时间 7 天），召回结果经过滤后再进入排序。</p>
<h3>6.3 短视频流的特殊处理</h3>
<p>短视频流（类似抖音的沉浸式上下滑）和图文信息流在分发层有几个关键差异：</p>

<h3>6.5 SLA 设计</h3>
<p>Feed 接口的 SLA 需要区分<strong>模型链路延迟</strong>和<strong>端到端延迟</strong>：</p>

<td>召回 + 粗排 + 精排 + 重排</td>
</tr>

<td><strong>Feed 接口 P50</strong></td>
<td>&lt; 80ms</td>
<td>含模型链路 + 服务逻辑 + 序列化</td>

<tr>
<td><strong>Feed 接口 P99</strong></td>
<td>&lt; 300ms</td>

<tr>
<td><strong>可用性</strong></td>
<td>99.95%</td>
<td>月度停机 &lt; 22 分钟</td>

<p><strong>容量规划</strong>：按峰值 QPS 的 2 倍预留资源。百万 DAU 的社区，Feed 接口峰值 QPS 通常在 5,000-20,000（取决于用户活跃度和刷新频率），需要 10-30 台应用服务器（按单机 1,000 QPS 估算）。</p>
<p><strong>一句话总结</strong>：Feed 分发层的核心是&quot;快&quot;和&quot;稳&quot;——正常情况下要快，异常情况下不能挂。</p>

<h2>7. 搜索系统</h2>
<p>搜索是信息流之外的<strong>第二大流量入口</strong>。小红书的搜索流量占比已超 30%，抖音搜索的日均使用次数超过 5 亿。搜索和推荐共享内容理解和特征基础设施，但在召回和排序上有本质差异。</p>

<h3>7.2 跨场景意图传递</h3>
<p>搜索行为是推荐的强意图信号——用户搜了&quot;露营装备&quot;，推荐流应该在接下来的 Session 中增加户外类内容的权重。反过来，用户在推荐流中持续消费美食内容，搜索联想也应优先展示美食相关词。</p>
<p><strong>实时联动的技术实现</strong>：搜索系统把用户的 Query 关键词实时写入行为序列（Kafka → Flink → Redis），推荐的精排模型通过 DIN/DIEN 的 Attention 机制自动捕捉这些跨场景信号。关键是延迟——从用户搜索到推荐流感知到这个意图，需要控制在<strong>分钟级</strong>。</p>
<p>跨场景一致性的另一面是<strong>负反馈共享</strong>：用户在推荐流对某类内容点了&quot;不感兴趣&quot;，搜索结果中同类内容也应降权。这需要统一的用户偏好存储，而非各场景维护独立的负反馈列表。</p>

</tbody></table>
<p>这些播放体验指标不直接进入推荐模型，但会显著影响完播率和留存——首帧慢 100ms，完播率下降 2-3%。</p>
<h3>10.2 AB 实验平台</h3>
<p>AB 实验是推荐系统迭代的基础设施——每一次召回策略调整、模型升级、重排规则变更都必须通过 AB 实验验证。</p>
<p><strong>核心架构</strong>：</p>
<table>
<thead>
<tr>
<th>组件</th>
<th>职责</th>
<th>实现方案</th>
</tr>
</thead>
<tbody><tr>
<td><strong>分流服务</strong></td>
<td>将用户随机分到实验组/对照组</td>
<td>用户 ID Hash 取模，保证分流一致性</td>
</tr>
<tr>
<td><strong>实验配置</strong></td>
<td>管理实验参数、流量比例、互斥域</td>
<td>配置中心 + 管理后台</td>
</tr>
<tr>
<td><strong>指标收集</strong></td>
<td>收集各实验组的行为数据</td>
<td>Kafka → Flink → 实时指标聚合</td>
</tr>
<tr>
<td><strong>效果分析</strong></td>
<td>计算显著性、置信区间</td>
<td>T 检验 / Mann-Whitney U 检验</td>
</tr>
<tr>

<h3>10.3 推荐 Trace（全链路追踪）</h3>
<p>推荐系统的每一次请求，需要记录完整的决策过程，用于效果归因和问题排查。</p>
<p>一条推荐 Trace 应包含：</p>
<table>
<thead>
<tr>
<th>字段</th>
<th>内容</th>
<th>用途</th>
</tr>
</thead>
<tbody><tr>
<td><strong>request_id</strong></td>
<td>唯一请求标识</td>
<td>全链路关联</td>
</tr>
<tr>
<td><strong>user_features</strong></td>
<td>请求时使用的用户特征快照</td>
<td>排查特征问题</td>
</tr>
<tr>
<td><strong>recall_sources</strong></td>
<td>各路召回返回的内容列表和来源标记</td>
<td>分析召回贡献</td>
</tr>
<tr>
<td><strong>rank_scores</strong></td>
<td>粗排/精排/重排的打分结果</td>
<td>分析排序合理性</td>
</tr>
<tr>
<td><strong>final_list</strong></td>
<td>最终返回的内容列表</td>
<td>与用户行为对比</td>
</tr>
<tr>
<td><strong>latency_breakdown</strong></td>
<td>各阶段耗时</td>
<td>性能优化</td>
</tr>
<tr>
<td><strong>experiment_groups</strong></td>
<td>用户所在的实验组</td>
<td>AB 实验归因</td>
</tr>
</tbody></table>
<p>Trace 数据量大（每次请求一条，日均可达数亿条），通常存储在列式数据库（ClickHouse）中，保留 7-30 天。</p>
<p><strong>一句话总结</strong>：没有 AB 实验和全链路 Trace，推荐优化就是盲人摸象——你不知道改了什么导致了什么结果。</p>

<h2>总结</h2>
<p>信息流的技术架构是一条端到端的数据流水线——内容从创建、理解、召回、排序到分发，每一层都在做减法和排序。这条流水线的质量取决于各环节的协同效率，而非某个组件有多厉害。</p>
<p>三个核心原则：<strong>读写分离</strong>保证写入慢不影响体验，<strong>三级流水线</strong>（离线/近线/在线）平衡效果和延迟，<strong>度量驱动优化</strong>确保每一次迭代都有数据支撑。</p>
<p>技术架构解决的是&quot;能不能&quot;——系统能不能在 200ms 内返回用户感兴趣的内容。而&quot;好不好&quot;——生态是否健康、商业化是否可持续、团队如何演进——这些问题，留给下篇<a href="/blog/engineering/architecture/%E7%99%BE%E4%B8%87%E7%BA%A7%E5%86%85%E5%AE%B9%E7%A4%BE%E5%8C%BA%E4%BF%A1%E6%81%AF%E6%B5%81%EF%BC%88%E4%B8%8B%EF%BC%89%EF%BC%9A%E8%BF%90%E8%90%A5%E4%BD%93%E7%B3%BB%E4%B8%8E%E5%95%86%E4%B8%9A%E5%8C%96">《百万级内容社区信息流（下）：运营体系与商业化》</a>。</p>
